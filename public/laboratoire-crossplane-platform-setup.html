Objectifs d'apprentissage

  • Comprendre l'architecture Crossplane : Control Plane, Provider, MR, XR, XRC.
  • Installer Crossplane via Helm sur un cluster Kubernetes local (minikube).
  • Installer et configurer un Provider AWS avec ses credentials.
  • Créer et observer une Managed Resource (Bucket S3) dans AWS.
  • Définir un CompositeResourceDefinition (XRD) avec un schéma OpenAPI.
  • Construire une Composition qui orchestre plusieurs Managed Resources.
  • Créer un Claim (XRC) en tant qu'utilisateur de la plateforme.
  • Déboguer les ressources Crossplane avec kubectl describe et les événements.

Concepts clés Crossplane

01
Provider
Composant qui parle à un cloud (AWS, GCP, Azure). Il expose des CRDs pour chaque ressource cloud sous-jacente.
provider-aws-s3
02
Managed Resource
Représentation 1:1 d'une ressource cloud réelle. Créer un Bucket Crossplane crée un vrai bucket S3.
Bucket.s3.aws.crossplane.io
03
XRD & XR
Le CompositeResourceDefinition définit un nouveau type API. La Composite Resource en est une instance cluster-scoped.
XPostgreSQLInstance
04
Composition
Définit comment un XR est implémenté : quelles MRs créer, quels patches appliquer.
Composition.apiextensions.v1
05
Claim (XRC)
Vue namespace-scoped pour les développeurs. Un Claim « réserve » un XR sans exposer la complexité cloud.
PostgreSQLInstance (claim)

Déroulé du laboratoire

  1. Préparer l'environnement

    Démarrez minikube avec au moins 4 Go de RAM et 2 vCPUs. Vérifiez que kubectl et helm sont installés et pointent vers le bon contexte.

    Le lab fonctionne aussi sur kind ou k3d. Minikube est recommandé pour sa simplicité.

  2. Installer Crossplane

    Ajoutez le repo Helm crossplane-stable, installez Crossplane dans le namespace crossplane-system et attendez que les 2 pods (crossplane et crossplane-rbac-manager) soient Running.

  3. Installer le Provider AWS

    Appliquez un manifest Provider pointant vers provider-aws-s3. Crossplane télécharge le package OCI et installe les CRDs AWS S3 dans le cluster (environs 90 secondes).

  4. Configurer les credentials AWS

    Créez un Secret Kubernetes contenant votre Access Key et Secret Key AWS, puis liez-le au Provider via un ProviderConfig utilisant source: Secret.

    En production, utilisez le provider d'identité IRSA (IAM Role for Service Account) ou Upbound Cloud Credentials — évitez les credentials statiques.

  5. Créer une Managed Resource S3

    Appliquez un manifest Bucket (genre s3.aws.crossplane.io/v1beta1) avec forProvider.region: eu-central-1. Observez la création dans la console AWS et vérifiez les conditions READY et SYNCED.

  6. Définir un XRD et une Composition

    Créez un CompositeResourceDefinition nommant une ressource XStorageBucket avec son Claim StorageBucket. Puis créez une Composition qui orchestre une Bucket MR à partir de l'XR, avec un patch pour injecter le nom depuis le Claim.

  7. Créer un Claim et valider

    Dans le namespace team-a, appliquez un StorageBucket Claim. Crossplane crée automatiquement l'XR et la Bucket MR correspondante. Vérifiez avec kubectl get managed et kubectl describe storagebucket.

Points de validation

Tous les pods crossplane-system sont en état Running.
Le Provider provider-aws-s3 affiche INSTALLED=True et HEALTHY=True.
La Managed Resource Bucket affiche READY=True et SYNCED=True.
Le bucket existe réellement dans la console AWS (ou localstack) avec les bons tags.
Le Claim StorageBucket dans team-a est READY=True et bound à un XR.
kubectl get managed liste toutes les MRs avec leur statut SYNCED.
Modifier le Claim (ex. changer un tag) se propage automatiquement à la ressource cloud.
