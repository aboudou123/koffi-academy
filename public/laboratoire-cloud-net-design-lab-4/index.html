Instructions

Architecture de sécurité

Vous concevez la sécurité réseau du VPC production : des security groups par tier (pare-feu au niveau instance, à état) et des Network ACLs (filtrage au niveau sous-réseau, sans état) pour une défense en profondeur. Chaque tier n'accepte que le trafic du tier précédent.

Progression0 / 4

Étape 1 · Security groups par tier

Dans l'onglet Design Files, créez security_design.yml — un security group par tier, chacun n'acceptant que le tier précédent : 
# TaskFlow Production Security Architecture
vpc: "taskflow-prod-vpc"   # 10.2.0.0/16

security_groups:
  web_sg:
    description: "Web tier — public load balancers"
    ingress:
      - { proto: tcp, port: 443, source: "0.0.0.0/0" }
      - { proto: tcp, port: 80,  source: "0.0.0.0/0" }
    egress:
      - { proto: tcp, port: 8080, dest: "app_sg" }
  app_sg:
    description: "App tier — accepts only the web tier"
    ingress:
      - { proto: tcp, port: 8080, source: "web_sg" }
    egress:
      - { proto: tcp, port: 5432, dest: "db_sg" }
  db_sg:
    description: "Database tier — accepts only the app tier"
    ingress:
      - { proto: tcp, port: 5432, source: "app_sg" }
    egress: []   # no outbound to internet

Étape 2 · Network ACLs

Ajoutez les Network ACLs (append à security_design.yml) — filtrage au niveau sous-réseau, deuxième couche de défense : 
network_acls:
  public_nacl:    # web subnets
    inbound:
      - { rule: 100, proto: tcp, port: 443, source: "0.0.0.0/0", action: allow }
      - { rule: 110, proto: tcp, port: 80,  source: "0.0.0.0/0", action: allow }
      - { rule: "*", action: deny }
  private_app_nacl:
    inbound:
      - { rule: 100, proto: tcp, port: 8080, source: "10.2.0.0/16", action: allow }
      - { rule: "*", action: deny }
  private_db_nacl:
    inbound:
      - { rule: 100, proto: tcp, port: 5432, source: "10.2.11.0/24", action: allow }
      - { rule: 110, proto: tcp, port: 5432, source: "10.2.12.0/24", action: allow }
      - { rule: "*", action: deny }

Étape 3 · Valider la sécurité

Vérifiez dans le Terminal que chaque tier n'accepte que le précédent et que la db est isolée : 
/root/vpc-design/scripts/validate_security.sh security_design.yml

Étape 4 · Justifier la sécurité

Ajoutez la justification de la défense en profondeur à design_rationale.md (onglet Design Files) : 
## Architecture de securite

Decision: security groups par tier, chaque tier n'accepte que le precedent.
Rationale: principe du moindre privilege, segmentation du trafic est-ouest,
seul le web accepte 443/80 depuis Internet.

Decision: Network ACLs en complement des security groups.
Rationale: defense en profondeur — filtrage subnet (sans etat) en plus
du pare-feu instance (a etat) ; regle implicite deny en fin de liste.

Decision: le tier db n'a aucune sortie (egress vide) et n'accepte que 5432
depuis le tier app. Rationale: surface d'attaque minimale pour les donnees.

Critères de réussite

  • 3 security groups (web/app/db) définis.
  • Network ACLs par sous-réseau ajoutés.
  • Sécurité validée (chaînage des tiers, db isolée).
  • Justification de sécurité documentée.

Astuce : tapez check dans le terminal pour vérifier votre progression.
