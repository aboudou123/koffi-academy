Instructions

Créer une politique d'autorisation

La baseline default-deny du chapitre 2 bloque tout. Vous allez maintenant rouvrir sélectivement frontend → backend (port 80) avec une CiliumNetworkPolicy, vérifier que ce flux passe, et confirmer que les autres pods restent bloqués.

Progression0 / 4

Étape 1 · Créer et appliquer la politique d'autorisation

Créez une CiliumNetworkPolicy qui autorise frontend → backend sur le port 80/TCP. Dans le Terminal (here-document) ou l'onglet Policies : 
cat > /root/cilium/policies/allow-frontend-to-backend.yaml << 'EOF'
apiVersion: cilium.io/v2
kind: CiliumNetworkPolicy
metadata:
  name: allow-frontend-to-backend
  namespace: policy-demo
spec:
  endpointSelector:
    matchLabels:
      app: backend
  ingress:
  - fromEndpoints:
    - matchLabels:
        app: frontend
    toPorts:
    - ports:
      - port: "80"
        protocol: TCP
EOF
CiliumNetworkPolicy (API cilium.io/v2) est plus puissante qu'une NetworkPolicy standard : identités par labels, filtrage L7 et application eBPF. Ici, endpointSelector cible le backend, fromEndpoints autorise le frontend, toPorts limite au port 80/TCP.
Appliquez la politique dans le Terminal : 
kubectl apply -f /root/cilium/policies/allow-frontend-to-backend.yaml

Étape 2 · Tester le trafic autorisé

Récupérez l'IP du backend et testez la connexion depuis le frontend (maintenant autorisée) : 
BACKEND_IP=$(kubectl get pod -n policy-demo -l app=backend -o jsonpath='{.items[0].status.podIP}')
kubectl exec -n policy-demo deploy/frontend -- curl -s --connect-timeout 5 http://$BACKEND_IP
Vous devez voir la page nginx : le flux frontend → backend est rétabli. La politique est précise (port 80 uniquement) : un autre port reste bloqué : 
kubectl exec -n policy-demo deploy/frontend -- curl -s --connect-timeout 5 http://$BACKEND_IP:8080 || echo "Port 8080 blocked as expected"

Étape 3 · Vérifier que les autres trafics sont bloqués

Créez un pod de test (sans le label app: frontend) et attendez qu'il soit prêt : 
kubectl run test-pod -n policy-demo --image=curlimages/curl:8.11.1 --restart=Never --command -- sleep 3600
kubectl wait --for=condition=Ready pod/test-pod -n policy-demo --timeout=60s
Testez depuis ce pod : il n'a pas de règle d'autorisation, donc le trafic est bloqué : 
kubectl exec -n policy-demo test-pod -- curl -s --connect-timeout 5 http://$BACKEND_IP || echo "Connection blocked as expected"

Étape 4 · Lister et inspecter les politiques

Listez les politiques (standard + Cilium), puis inspectez la politique d'autorisation : 
kubectl get networkpolicies -n policy-demo && echo "---" && kubectl get ciliumnetworkpolicies -n policy-demo
kubectl describe ciliumnetworkpolicy allow-frontend-to-backend -n policy-demo
Dans la sortie de describe, Enforcing: true confirme que la politique est activement appliquée.

Critères de réussite

  • Politique d'autorisation créée (allow-frontend-to-backend).
  • Le frontend atteint le backend (port 80).
  • Les autres pods (test-pod) restent bloqués.
  • La politique est listée et activement appliquée.
