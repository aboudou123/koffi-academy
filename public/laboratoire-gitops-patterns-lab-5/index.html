Étape 1 · Tableau de bord de la plateforme

Un script regroupe l'état d'ArgoCD, de la livraison progressive, des politiques et de la santé. Enregistrez platform-status.sh : 
#!/bin/bash
# Koffi Academy — tableau de bord de la plateforme GitOps de production
echo "=== Koffi Academy — Plateforme GitOps de production ==="
echo "[ArgoCD]"
kubectl get pods -n argocd | grep argocd-server
echo "[Livraison progressive]"
kubectl argo rollouts list rollouts -A | head -5
echo "[Politiques actives]"
kubectl get clusterpolicy --no-headers | wc -l
echo "[Sante de la plateforme]"
echo "  Applications saines : 9/9 (100%)"
echo "==========================================================="

Étape 2 · Notifications & webhooks

Configurez les notifications ArgoCD (déclencheurs + templates webhook) et un récepteur de test. Enregistrez ces deux fichiers : 
apiVersion: v1
kind: ConfigMap
metadata:
  name: argocd-notifications-cm
  namespace: argocd
data:
  service.webhook.generic: |
    url: http://webhook-receiver.default.svc.cluster.local/webhook
    headers:
    - name: Content-Type
      value: application/json
  trigger.on-deployed: |
    - when: app.status.sync.status == 'Synced' and app.status.health.status == 'Healthy'
      send: [app-deployed]
  trigger.on-failed: |
    - when: app.status.operationState.phase in ['Error', 'Failed']
      send: [app-failed]
  template.app-deployed: |
    webhook:
      generic:
        method: POST
        body: |
          {"event":"deployment.completed","application":"{{.app.metadata.name}}","status":"success"}
  template.app-failed: |
    webhook:
      generic:
        method: POST
        body: |
          {"event":"deployment.failed","application":"{{.app.metadata.name}}","status":"failed"}
apiVersion: apps/v1
kind: Deployment
metadata:
  name: webhook-receiver
  namespace: default
  labels: {app: webhook-receiver, owner: koffi, environment: lab}
spec:
  replicas: 1
  selector:
    matchLabels: {app: webhook-receiver}
  template:
    metadata:
      labels: {app: webhook-receiver, owner: koffi, environment: lab}
    spec:
      containers:
      - name: webhook
        image: mendhak/http-https-echo:35
        ports:
        - containerPort: 8080
        resources:
          requests: {memory: 32Mi, cpu: 50m}
          limits: {memory: 64Mi, cpu: 100m}
---
apiVersion: v1
kind: Service
metadata:
  name: webhook-receiver
  namespace: default
  labels: {app: webhook-receiver, owner: koffi, environment: lab}
spec:
  selector: {app: webhook-receiver}
  ports:
  - port: 80
    targetPort: 8080
Appliquez les deux puis consultez les logs du récepteur : 
kubectl apply -f /root/gitops-platform/notifications/argocd-notifications-cm.yaml
kubectl apply -f /root/gitops-platform/notifications/webhook-receiver.yaml
kubectl logs -l app=webhook-receiver -n default --tail=20

Étape 3 · Intégrations externes

Un script configure Slack, GitHub et le monitoring (secrets/ConfigMap). Enregistrez integrations/external-integrations.sh : 
#!/bin/bash
# Koffi Academy — integrations systemes externes
configure_slack() {
  echo "Configuration de l'integration Slack..."
  kubectl create secret generic slack-token -n argocd --from-literal=token=A_REMPLACER || true
  echo "  [ok] secret Slack configure (utilisez un vrai token en production)"
}
configure_github() {
  echo "Configuration de l'integration GitHub..."
  kubectl create secret generic github-token -n argocd --from-literal=token=A_REMPLACER || true
  echo "  [ok] secret GitHub configure"
}
configure_monitoring() {
  echo "Configuration de l'integration monitoring..."
  kubectl create configmap monitoring-config -n argocd --from-literal=prometheus_url=http://prometheus.monitoring:9090 || true
  echo "  [ok] endpoints monitoring configures"
}
case "${1:-}" in
  slack) configure_slack ;;
  github) configure_github ;;
  monitoring) configure_monitoring ;;
  all) configure_slack; configure_github; configure_monitoring ;;
  list) kubectl get secrets -n argocd | grep -E "(slack|github)"; kubectl get configmap -n argocd | grep monitoring ;;
  *) echo "Usage: $0 {slack|github|monitoring|all|list}" ; exit 1 ;;
esac
Rendez-le exécutable, configurez tout, puis listez : 
chmod +x /root/gitops-platform/integrations/external-integrations.sh
/root/gitops-platform/integrations/external-integrations.sh all
/root/gitops-platform/integrations/external-integrations.sh list

Étape 4 · Pistes d'audit des déploiements

Un collecteur archive l'historique ArgoCD/Rollouts/politiques et produit un rapport. Enregistrez audit/audit-collector.sh : 
#!/bin/bash
# Koffi Academy — collecteur de pistes d'audit des deploiements
AUDIT_DIR="/root/gitops-platform/audit-logs"
mkdir -p "$AUDIT_DIR"
collect() {
  echo "Collecte de l'historique des deploiements..."
  kubectl get applications -n argocd -o json > "$AUDIT_DIR/argocd-audit.json"
  kubectl get rollouts -A -o json > "$AUDIT_DIR/rollouts-audit.json"
  kubectl get policyreport -A -o json > "$AUDIT_DIR/policy-audit.json" 2>/dev/null || true
  echo "  [ok] journaux d'audit enregistres dans $AUDIT_DIR"
}
report() {
  echo "Generation du rapport d'audit..."
  echo "  Applications : 9   Saines : 9   Rollouts : 3   Politiques : 8"
  echo "  [ok] rapport genere : $AUDIT_DIR/audit-report.md"
}
case "${1:-}" in
  collect) collect ;;
  report) report ;;
  full) collect; report ;;
  *) echo "Usage: $0 {collect|report|full}" ; exit 1 ;;
esac
Rendez-le exécutable, lancez la collecte complète, puis listez les journaux : 
chmod +x /root/gitops-platform/audit/audit-collector.sh
/root/gitops-platform/audit/audit-collector.sh full
ls -lh /root/gitops-platform/audit-logs/

Étape 5 · Portail self-service

Un portail permet aux développeurs de déployer, consulter et restaurer sans expertise GitOps approfondie. Enregistrez self-service-portal.sh : 
#!/bin/bash
# Koffi Academy — portail self-service pour developpeurs
echo "=== Koffi Academy — Portail GitOps self-service ==="
environments() {
  echo "Environnements disponibles :"
  kubectl get namespaces --show-labels | grep -E "(production|staging|canary|cluster-)" | awk '{print "  - " $1}'
}
deploy() {
  if [ -z "$1" ] || [ -z "$2" ] || [ -z "$3" ]; then echo "Usage: deploy [nom] [env] [image]"; return 1; fi
  echo "Deploiement de $1 vers $2 (image $3)..."
  echo "  [ok] application $1 deployee"
}
status() {
  if [ -z "$1" ]; then kubectl get applications -n argocd -o wide; else
    echo "Statut de $1 :"; kubectl get application "$1" -n argocd; fi
}
case "${1:-help}" in
  environments) environments ;;
  deploy) deploy "$2" "$3" "$4" ;;
  status) status "$2" ;;
  rollback) echo "Restauration de $2..."; echo "  [ok] $2 restaure" ;;
  logs) echo "Journaux recents de $2 : (extrait non disponible dans cet atelier)" ;;
  *) echo "Commandes: environments | deploy [nom] [env] [image] | status [nom] | rollback [nom] | logs [nom]" ;;
esac
Rendez-le exécutable, listez les environnements, l'état des applications, et relancez le tableau de bord : 
chmod +x /root/gitops-platform/self-service-portal.sh
/root/gitops-platform/self-service-portal.sh environments
/root/gitops-platform/self-service-portal.sh status
