Prerequis et posture de travail

Ce cours part du principe que vous savez deja ecrire une ressource Terraform simple. Le niveau avance commence lorsque plusieurs personnes, plusieurs comptes AWS et plusieurs environnements doivent cohabiter sans casser la production.

Terraform

Variables, outputs, data sources, for_each, modules, plan/apply/destroy.

AWS

IAM, VPC, subnets, security groups, S3, KMS, CloudWatch, notion de compte et region.

Git

Branches, pull requests, revue de code, tags et workflow CI/CD minimal.

Regle de cours: toute infrastructure doit pouvoir etre reconstruite a partir de Git, d'un state connu et d'identites AWS temporaires. Les clics console sont reserves au diagnostic, pas au changement durable.

Structure de depot recommandee

Separez les modules reutilisables des stacks deployables. Un module decrit un pattern. Une stack instancie ce pattern pour un environnement donne.

Architecture cible

Le projet final construit une fondation AWS typique: state centralise, reseau multi-AZ, roles CI/CD, services prives, controles de qualite et traces d'audit. Le but n'est pas d'empiler des services, mais de comprendre les decisions qui rendent une plateforme exploitable.

Flux cible

  • Le code Terraform vit dans Git et passe par revue.
  • La pipeline assume un role AWS via OIDC, sans secret statique.
  • Le state est stocke dans S3 avec versioning, chiffrement et verrouillage.
  • Les stacks sont separees par domaine: network, platform, data.
1

Compte AWS, bucket state, KMS, politique d'acces et role CI.

3

Module 1

Provider AWS, versions et conventions

Le provider n'est pas un detail. C'est le contrat entre votre code et l'API AWS. En production, vous devez le versionner, limiter les comptes autorises, imposer des tags et rendre l'authentification explicite.

Fichiers de base

versions.tfhcl
terraform {
  required_version = ">= 1.10.0"

  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }
}
providers.tfhcl
provider "aws" {
  region              = var.aws_region
  allowed_account_ids = [var.aws_account_id]

  default_tags {
    tags = {
      ManagedBy   = "Terraform"
      Project     = var.project
      Environment = var.environment
      Owner       = var.owner
    }
  }
}
Erreur classique: utiliser le profil AWS local d'un engineer comme mecanisme de production. La bonne cible est un role assumable avec des permissions courtes, auditees et separees par environnement.
Module 2

State distant S3 et verrouillage

Le state Terraform contient des donnees sensibles et represente la verite operationnelle de votre infrastructure. Il doit etre prive, chiffre, versionne, verrouille et separe par stack.

backend.tfhcl
terraform {
  backend "s3" {
    bucket       = "koffi-terraform-state-prod"
    key          = "network/prod/terraform.tfstate"
    region       = "eu-central-1"
    encrypt      = true
    use_lockfile = true
  }
}

Le verrouillage S3 avec use_lockfile evite deux apply simultanes sur le meme state. Pour les anciens projets, vous pouvez encore rencontrer un verrouillage DynamoDB; considerez-le comme heritage et planifiez la migration.

Commandes de migration

migrationbash
terraform init
terraform plan
terraform init -migrate-state
terraform state list
Bucket S3 avec versioning active.
Chiffrement SSE-S3 ou KMS selon la criticite.
Acces limite aux roles Terraform seulement.
Chemins de state separes par environnement.
Module 3

VPC multi-AZ et patterns reseau

Un reseau Terraform avance doit rendre les decisions visibles: plage CIDR, segmentation, exposition internet, routage prive, endpoints et cout NAT. Le code doit empecher les architectures ambiguës.

network stackhcl
module "network" {
  source = "../../../modules/network"

  name               = "koffi-prod"
  cidr_block         = "10.40.0.0/16"
  availability_zones = ["eu-central-1a", "eu-central-1b", "eu-central-1c"]

  public_subnet_bits  = 8
  private_subnet_bits = 8
  enable_nat_gateway  = true
  single_nat_gateway  = false
}

Principes de conception

  • Un subnet public contient uniquement ce qui doit recevoir une route vers l'internet gateway.
  • Les workloads applicatifs, bases de donnees et runners restent en subnets prives.
  • Les endpoints VPC reduisent la dependance au NAT pour S3, ECR, CloudWatch et STS.
  • La haute disponibilite exige une repartition sur plusieurs AZ, pas seulement plusieurs subnets.
Regle reseau: une ressource n'est pas "privee" parce que son nom contient private. Elle est privee si ses routes, security groups, NACL, DNS et dependances ne l'exposent pas.
Module 4

Modules reutilisables et interfaces strictes

Un bon module n'est pas un dossier qui cache du code. C'est une interface stable qui encode une decision d'architecture. Plus le module est generique, moins il protege. Plus il est explicite, plus il devient exploitable.

variables.tfhcl
variable "subnets" {
  description = "Subnets a creer par zone de disponibilite."
  type = map(object({
    cidr_block        = string
    availability_zone = string
    public            = bool
  }))

  validation {
    condition     = length(var.subnets) >= 2
    error_message = "Au moins deux subnets sont requis pour une architecture multi-AZ."
  }
}
main.tfhcl
resource "aws_subnet" "this" {
  for_each = var.subnets

  vpc_id            = aws_vpc.this.id
  cidr_block        = each.value.cidr_block
  availability_zone = each.value.availability_zone

  map_public_ip_on_launch = each.value.public
}

Checklist module production

  • Inputs typés, avec validations utiles et valeurs par defaut rares.
  • Outputs explicites, stables et documentes.
  • Aucun provider cache dans le module sauf cas justifie.
  • Pas de logique metier implicite basee sur des noms magiques.
Module 5

IAM, federation OIDC et moindre privilege

La maturite Terraform se voit dans IAM. Le but est simple: aucun secret AWS longue duree dans la CI, des roles par environnement, des permissions minimales et une trace CloudTrail exploitable.

github-oidc-role.tfhcl
data "aws_iam_policy_document" "github_oidc_assume" {
  statement {
    actions = ["sts:AssumeRoleWithWebIdentity"]

    principals {
      type        = "Federated"
      identifiers = [aws_iam_openid_connect_provider.github.arn]
    }

    condition {
      test     = "StringEquals"
      variable = "token.actions.githubusercontent.com:aud"
      values   = ["sts.amazonaws.com"]
    }

    condition {
      test     = "StringLike"
      variable = "token.actions.githubusercontent.com:sub"
      values   = ["repo:koffi-academy/terraform-aws-platform:ref:refs/heads/main"]
    }
  }
}

resource "aws_iam_role" "terraform_ci" {
  name               = "terraform-ci-prod"
  assume_role_policy = data.aws_iam_policy_document.github_oidc_assume.json
}
Point de rigueur: commencez avec une permission plus large uniquement pendant le design initial, puis capturez les actions reelles via CloudTrail et reduisez la policy avant de declarer le module pret pour production.
Module 6

Compute, EKS, RDS et donnees sensibles

Les services managés AWS simplifient l'exploitation, mais ne suppriment pas la responsabilite d'architecture. Terraform doit exprimer le niveau de disponibilite, de sauvegarde, de chiffrement et d'observabilite attendu.

EKS

Cluster prive, node groups separes, IRSA, logs control plane, add-ons versionnes et upgrades planifies.

RDS

Subnets prives, Multi-AZ selon criticite, KMS, deletion protection, backup window et monitoring.

rds-postgres.tfhcl
module "database" {
  source = "../../../modules/rds-postgres"

  name                 = "koffi-prod"
  engine_version       = "16"
  instance_class       = "db.m7g.large"
  subnet_ids           = module.network.private_subnet_ids
  security_group_ids   = [module.network.database_security_group_id]
  storage_encrypted    = true
  deletion_protection  = true
  backup_retention_days = 14
}

Ne mettez jamais un mot de passe en clair dans une variable Terraform versionnee. Utilisez AWS Secrets Manager, SSM Parameter Store, un flux Vault ou un mecanisme de rotation controle.

Module 7

CI/CD Terraform: plan, review, apply

Une pipeline Terraform n'a pas pour mission de deployer le plus vite possible. Elle doit rendre le changement lisible, verifier les risques et appliquer uniquement quand le contexte est clair.

.github/workflows/terraform.ymlyaml
name: terraform

on:
  pull_request:
    paths: ["stacks/**", "modules/**"]
  push:
    branches: ["main"]
    paths: ["stacks/**", "modules/**"]

permissions:
  id-token: write
  contents: read
  pull-requests: write

jobs:
  plan:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: hashicorp/setup-terraform@v3
      - uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: arn:aws:iam::123456789012:role/terraform-ci-prod
          aws-region: eu-central-1
      - run: terraform -chdir=stacks/prod/network fmt -check
      - run: terraform -chdir=stacks/prod/network init
      - run: terraform -chdir=stacks/prod/network validate
      - run: terraform -chdir=stacks/prod/network plan -out=tfplan

Politique d'apply

  • Pull request: format, init, validate, plan, scan securite.
  • Main branch: plan final, approbation humaine pour prod, apply avec role dedie.
  • Plan stocke comme artefact sensible avec retention courte.
  • Drift detection planifiee hors heures critiques avec notification, pas apply automatique aveugle.
Module 8

Gouvernance, tests et operations

Le Terraform avance n'est pas seulement syntaxique. Il inclut des garde-fous: validations, preconditions, checks, tests, policies, tagging, couts et runbooks.

checks.tfhcl
check "private_database_subnets" {
  assert {
    condition = alltrue([
      for subnet in module.network.database_subnets : subnet.map_public_ip_on_launch == false
    ])
    error_message = "Les subnets database ne doivent pas attribuer d'IP publique."
  }
}
Anti-patternRisqueCorrection
Un seul state pour toutBlast radius enorme et plans illisibles.Stacks separees par domaine et environnement.
Modules trop generiquesInterfaces floues, changements dangereux.Modules orientes pattern avec validations.
Apply depuis poste localIdentite non auditee, drift de version, permissions floues.Apply via pipeline et roles federes.
Secrets dans tfvarsFuite Git, state sensible, rotation difficile.Secrets Manager, SSM, Vault ou injection runtime.
Discipline operationnelle: chaque module critique doit avoir un runbook minimal: comment planifier, appliquer, annuler, restaurer le state et diagnostiquer une erreur AWS.

Labs pratiques

Les labs sont conçus pour forcer les bonnes decisions. Vous produisez du code, un plan, une preuve de controle et une note d'exploitation.

1

Bootstrap state

Creer le bucket S3 de state, activer versioning, chiffrement, verrouillage et policy IAM minimale.

  • Livrable: backend fonctionnel.
  • Verification: deux plans concurrents ne passent pas.
2

VPC multi-AZ

Construire un module network avec subnets publics/prives, NAT, endpoints et outputs consommables.

  • Livrable: module documente.
  • Verification: routage et tags corrects.
3

Role OIDC CI

Mettre en place l'identite federée pour GitHub Actions et supprimer toute cle AWS statique.

  • Livrable: role assumable.
  • Verification: CloudTrail montre l'assume-role.
4

RDS prive

Deployer une base PostgreSQL chiffree, protegee contre suppression, non exposee publiquement.

  • Livrable: module data.
  • Verification: pas d'IP publique, backup actif.
5

Pipeline Terraform

Executer fmt, init, validate, plan, scan et apply controle sur une stack prod simulee.

  • Livrable: workflow CI.
  • Verification: PR avec plan lisible.
6

Drift et incident

Modifier une ressource hors Terraform, detecter le drift, choisir entre import, revert ou changement de code.

  • Livrable: rapport d'incident.
  • Verification: decision justifiee.

Evaluation finale

Le projet final consiste a livrer une fondation Terraform AWS prete pour revue d'architecture. Le resultat attendu n'est pas seulement un apply reussi, mais un systeme comprehensible et defendable.

Le state est separe, verrouille, versionne et chiffre.
Les modules ont des inputs types, validations et outputs stables.
La CI utilise OIDC et publie un plan lisible en pull request.
Le reseau distingue clairement exposition publique et privee.
Les ressources critiques sont taguees, chiffrees et monitorables.
Un runbook explique apply, rollback, drift et restauration state.

Livrables attendus

  • Un repository Terraform separe entre modules et stacks, avec README par module.
  • Un diagramme d'architecture indiquant comptes, roles, state, reseau, donnees et flux CI/CD.
  • Un journal de decisions qui justifie les compromis cout, securite, disponibilite et complexite.
  • Un plan de reprise: restauration du state, import de ressources, rollback applicatif et gestion du drift.

Runbook d'exploitation Terraform AWS

Un cours avance doit vous laisser avec une methode exploitable apres la formation. Ce runbook resume les reflexes attendus pour une equipe qui gere une infrastructure critique avec Terraform.

SituationAction recommandeePreuve attendue
Changement normalPull request, plan commente, validation securite, approbation, apply via role CI.PR, plan conserve, logs CI et CloudTrail.
Drift detecteComparer Git, state et ressource AWS. Decider: revert manuel, import, ou changement Terraform.Ticket d'incident avec decision et plan final propre.
State corrompu ou bloqueStopper les applies, sauvegarder le state, identifier le lock, restaurer ou reparer avec deux validateurs.Backup state, historique S3, journal de restauration.
Erreur provider AWSVerifier quota, region, version provider, permissions IAM et eventual consistency AWS.Message d'erreur, CloudTrail, version lock file.
RollbackRevenir par Git, pas par console. Rejouer plan et apply sauf urgence documentee.Commit de rollback, plan, validation post-apply.
Regle d'incident: ne supprimez jamais un state ou un lock sans preuve de l'operation en cours. En production, la rapidite vient de la preparation du runbook, pas de commandes improvisees.
