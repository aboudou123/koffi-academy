Was ist Argo CD — und warum sollten Sie es kennen?

Was ist Argo CD?

Argo CD ist ein deklarativer, GitOps-basierter Continuous-Delivery-Controller für Kubernetes. Es wurde 2018 von Intuit entwickelt, 2020 als Incubating-Projekt in die Cloud Native Computing Foundation (CNCF) aufgenommen und erreichte 2022 den Graduated-Status — das höchste Reifezeichen innerhalb der CNCF. Es zählt heute zu den meistgenutzten Deployment-Werkzeugen für Kubernetes weltweit.

Kernidee in einem Satz: Argo CD liest den gewünschten Zustand Ihrer Infrastruktur aus einem Git-Repository und sorgt dafür, dass Ihr Kubernetes-Cluster diesen Zustand jederzeit widerspiegelt — automatisch, nachvollziehbar und prüfbar.

Das Problem: Kubernetes-Deployments ohne Kontrolle

Um Argo CD zu verstehen, müssen wir zuerst das Problem verstehen, das es löst. Wie sahen Kubernetes-Deployments vor GitOps-Tools aus?

Traditioneller Ansatz (problematisch)

  • Entwickler oder Pipelines laufen direkt kubectl apply auf dem Cluster
  • CI-Pipelines haben direkten Schreibzugriff auf Kubernetes — ein massives Sicherheitsrisiko
  • Niemand weiß sicher, welche Manifest-Version wirklich aktiv ist
  • Manuelle Patches (kubectl edit) hinterlassen keine Spur
  • Drift: Was in Git steht, unterscheidet sich von dem, was im Cluster läuft
  • Rollbacks sind fehleranfällig und manuell

GitOps mit Argo CD (löst das Problem)

  • Git ist die einzige Quelle der Wahrheit — kein direkter Cluster-Zugriff mehr nötig
  • CI-Pipelines schreiben nur in Git, nie direkt in Kubernetes
  • Argo CD zieht Änderungen aus Git und wendet sie kontrolliert an (Pull-Modell)
  • Jede Änderung ist als Pull Request sichtbar, prüfbar und genehmigt
  • Drift wird sofort erkannt und angezeigt — automatische Korrektur optional
  • Rollback = ein git revert, das Argo CD sofort ausrollt

Was ist GitOps? Die vier Prinzipien

GitOps ist kein Produkt — es ist ein Betriebsmodell für Cloud-native Systeme, das von Weaveworks 2017 geprägt und später durch die OpenGitOps-Initiative der CNCF formalisiert wurde. Es basiert auf vier Kernprinzipien:

1. Deklarativ

Der gewünschte Systemzustand wird vollständig deklarativ beschrieben — nicht als Abfolge von Befehlen, sondern als Zielzustand. Kubernetes-Manifeste, Helm Charts und Kustomize-Overlays sind Beispiele für deklarative Definitionen.

2. Versioniert und unveränderlich

Git speichert den Zielzustand versioniert und unveränderlich. Jeder Commit ist ein Snapshot. Die vollständige Änderungshistorie ist jederzeit abrufbar — wer hat was wann geändert und warum (PR-Beschreibung).

3. Pull-basiert

Software-Agenten (hier: Argo CD) ziehen Änderungen aus Git, anstatt dass Pipelines sie in den Cluster drücken. Das eliminiert externe Schreibzugriffe auf den Cluster und reduziert die Angriffsoberfläche drastisch.

4. Kontinuierlich reconciliert

Argo CD beobachtet den Cluster-Zustand kontinuierlich. Weicht der Live-Zustand vom gewünschten Zustand in Git ab (Drift), erkennt Argo CD das sofort und kann automatisch oder auf Anforderung korrigieren.

The problem: Kubernetes deployments without control

To understand Argo CD, we first have to understand the problem it solves. What did Kubernetes deployments look like before GitOps tools?

Traditional approach (problematic)

  • Developers or pipelines run kubectl apply directly against the cluster

  • Manual patches (kubectl edit) leave no trace

GitOps with Argo CD (solves the problem)

  • Drift is detected and displayed immediately — automatic correction optional
  • Rollback = a single git revert that Argo CD rolls out instantly

What is GitOps? The four principles

GitOps is not a product — it is an operating model for cloud-native systems, coined by Weaveworks in 2017 and later formalized by the CNCF's OpenGitOps initiative. It rests on four core principles:

1. Declarative

The desired system state is described entirely declaratively — not as a sequence of commands, but as a target state. Kubernetes manifests, Helm charts and Kustomize overlays are examples of declarative definitions.

2. Versioned and immutable

Git stores the target state versioned and immutably. Every commit is a snapshot. The full change history is available at any time — who changed what, when and why (PR description).

3. Pull-based

Software agents (here: Argo CD) pull changes from Git rather than having pipelines push them into the cluster. This eliminates external write access to the cluster and drastically reduces the attack surface.

4. Continuously reconciled

Argo CD continuously observes the cluster state. If the live state diverges from the desired state in Git (drift), Argo CD detects it immediately and can correct it automatically or on demand.

1. Understanding GitOps properly

GitOps means: the desired state of your platform lives declaratively in Git. Kubernetes is no longer the place where you manually "click in" changes, but the place where a controller continuously enforces the state described in Git.

The working rule: Every functionally relevant change becomes visible as a pull request, is reviewed, merged and then rolled out by the GitOps controller.

CI pipeline

Builds artifacts, tests code, scans images and publishes versioned container images. CI must not change Kubernetes directly.

Argo CD

Reads Git, compares desired state with live state and synchronizes Kubernetes according to defined policies.

This makes the deployment process traceable: who approved a change, which manifest version is active and why a rollout was triggered is all visible in Git and Argo CD.

1. Bien comprendre GitOps

GitOps signifie : l'etat souhaite de votre plateforme reside de maniere declarative dans Git. Kubernetes n'est plus l'endroit ou vous "cliquez" manuellement vos changements, mais l'endroit ou un controleur applique en continu l'etat decrit dans Git.

La regle de travail : Chaque changement metier pertinent devient visible sous forme de pull request, est revise, fusionne, puis deploye par le controleur GitOps.

CI et CD GitOps sont des responsabilites distinctes

Pipeline CI

Construit les artefacts, teste le code, scanne les images et publie des images de conteneur versionnees. La CI ne doit pas modifier Kubernetes directement.

Argo CD

Lit Git, compare l'etat souhaite a l'etat live et synchronise Kubernetes selon des policies definies.

Le processus de deploiement devient ainsi tracable : qui a approuve un changement, quelle version de manifeste est active et pourquoi un deploiement a ete declenche, tout cela est visible dans Git et Argo CD.

5. Sync-Strategien bewusst auswählen

Argo CD synchronisiert nicht blind. Sie definieren, ob Sync manuell ausgelöst wird, ob Drift automatisch korrigiert wird und ob Ressourcen, die aus Git entfernt wurden, auch aus Kubernetes gelöscht werden sollen.

  clusterResourceWhitelist:
    - group: ""
      kind: Namespace
  namespaceResourceWhitelist:
    - group: apps
      kind: Deployment
    - group: ""
      kind: Service
    - group: networking.k8s.io
      kind: Ingress

Bien gerer les secrets

  • Ne committez jamais de secrets en clair dans Git, meme dans des depots prives.
  • Utilisez l'External Secrets Operator, Sealed Secrets ou SOPS avec une rotation de cles claire.
  • Separez la definition du secret, la valeur du secret et la politique d'acces.
  • Verifiez les pull requests avec du secret scanning et du policy-as-code.
Note sur l'App-of-Apps : Le pattern app-of-apps est puissant pour le bootstrap, mais doit rester sous controle des admins. Celui qui controle le parent peut influencer de nombreuses applications enfants.
