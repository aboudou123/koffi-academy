Instructions

Mettre en place une politique deny-all

Dans cette tâche, vous posez une politique « default-deny » comme baseline de sécurité : on vérifie d'abord que le trafic passe, puis on bloque tout l'ingress par défaut, et on confirme que la communication est coupée.

Progression0 / 4

Étape 1 · Tester la connectivité avant la politique

Vérifiez d'abord que les pods communiquent. Récupérez l'IP du pod backend dans le Terminal : 
BACKEND_IP=$(kubectl get pod -n policy-demo -l app=backend -o jsonpath='{.items[0].status.podIP}')
echo "Backend IP: $BACKEND_IP"
Testez la connexion du frontend vers le backend : 
kubectl exec -n policy-demo deploy/frontend -- curl -s --connect-timeout 5 http://$BACKEND_IP
Vous devez voir la page d'accueil nginx : la connectivité fonctionne.

Étape 2 · Créer la politique default-deny

Créez une politique qui refuse tout l'ingress par défaut (NetworkPolicy Kubernetes standard, appliquée par Cilium). Le plus simple : créez le fichier directement dans le Terminal via un here-document (ou collez le manifeste dans l'onglet Policies sous default-deny.yaml) : 
cat > /root/cilium/policies/default-deny.yaml << 'EOF'
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-ingress
  namespace: policy-demo
spec:
  podSelector: {}
  policyTypes:
  - Ingress
EOF
Note : un podSelector vide sélectionne tous les pods du namespace, et déclarer Ingress sans règle crée une baseline « tout refuser ». Cilium l'applique, et des règles d'autorisation (CiliumNetworkPolicy) pourront ensuite ouvrir le trafic voulu.
Appliquez la politique dans le Terminal : 
kubectl apply -f /root/cilium/policies/default-deny.yaml

Étape 3 · Vérifier que la politique est appliquée

Vérifiez que la politique existe dans le Terminal : 
kubectl get networkpolicies -n policy-demo

Étape 4 · Tester la connectivité après la politique

Testez que le trafic est maintenant bloqué : 
kubectl exec -n policy-demo deploy/frontend -- curl -s --connect-timeout 5 http://$BACKEND_IP || echo "Connection blocked as expected"
La connexion doit expirer ou échouer : la baseline default-deny fonctionne. Vérifications additionnelles : l'ingress est bloqué dans les deux sens, mais le trafic sortant (egress) reste autorisé : 
kubectl exec -n policy-demo deploy/backend -- curl -s --connect-timeout 5 http://10.0.0.28 || echo "Backend cannot reach frontend"
kubectl exec -n policy-demo deploy/frontend -- curl -s --connect-timeout 5 http://google.com || echo "Outbound blocked?"

Critères de réussite

  • Connectivité initiale vérifiée (frontend → backend).
  • Politique default-deny appliquée.
  • Le trafic entre pods est bloqué.

Dépannage

Si le trafic n'est pas bloqué, confirmez que la politique existe et inspectez-la : 
kubectl describe networkpolicy default-deny-ingress -n policy-demo
Vérifiez aussi que $BACKEND_IP est bien défini (Étape 1) avant de retester.
