Objectifs d'apprentissage

  • Comprendre le modèle d'identités basé sur les labels de Cilium.
  • Appliquer une baseline deny-all au niveau d'un namespace.
  • Autoriser un trafic précis avec des règles L3/L4.
  • Distinguer politiques d'ingress et d'egress.
  • Tester la connectivité entre pods et interpréter les blocages.
  • Observer les flux autorisés et refusés avec Hubble.
  • Restreindre les communications selon le moindre privilège.
  • Diagnostiquer une règle trop permissive ou trop stricte.

Environnement virtuel

Sandbox
cilium@koffi-lab: ~
# 1 · Appliquer une baseline deny-all sur le namespace
$ kubectl apply -f deny-all.yaml
ciliumnetworkpolicy.cilium.io/deny-all created

# 2 · Autoriser frontend -> backend sur le port 8080 (L3/L4)
$ kubectl apply -f allow-frontend.yaml
ciliumnetworkpolicy.cilium.io/allow-frontend created

# 3 · Tester la connectivité entre pods
$ kubectl exec frontend -- curl -s backend:8080
ok

# 4 · Observer les verdicts de flux avec Hubble
$ cilium hubble observe --verdict DROPPED
frontend -> db:5432   DROPPED   (policy)

Déroulé du laboratoire

  1. Inspecter l'état initial

    Vérifiez que Cilium est sain (cilium status) et que les pods demo communiquent sans restriction.

  2. Poser la baseline deny-all

    Appliquez une CiliumNetworkPolicy qui bloque tout le trafic du namespace par défaut.

  3. Autoriser le trafic nécessaire

    Ouvrez précisément frontend → backend sur le bon port avec une règle L3/L4 par labels.

  4. Maîtriser ingress et egress

    Limitez aussi le trafic sortant : autorisez backend → db, refusez le reste.

  5. Tester et valider

    Confirmez que les flux autorisés passent et que les flux non prévus sont bloqués.

  6. Observer avec Hubble

    Visualisez les verdicts FORWARDED et DROPPED pour auditer vos règles.
