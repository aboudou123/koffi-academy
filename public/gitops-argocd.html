Was ist Argo CD — und warum sollten Sie es kennen?

Bevor wir Argo CD installieren oder YAML schreiben, müssen wir verstehen, welches Problem Argo CD löst, wie es architektonisch aufgebaut ist und warum es heute zu den wichtigsten Werkzeugen in der Cloud-Native-Welt gehört. Diese Einführung legt das konzeptionelle Fundament für alle folgenden Module.

Was ist Argo CD?

Argo CD ist ein deklarativer, GitOps-basierter Continuous-Delivery-Controller für Kubernetes. Es wurde 2018 von Intuit entwickelt, 2020 als Incubating-Projekt in die Cloud Native Computing Foundation (CNCF) aufgenommen und erreichte 2022 den Graduated-Status — das höchste Reifezeichen innerhalb der CNCF. Es zählt heute zu den meistgenutzten Deployment-Werkzeugen für Kubernetes weltweit.

Kernidee in einem Satz: Argo CD liest den gewünschten Zustand Ihrer Infrastruktur aus einem Git-Repository und sorgt dafür, dass Ihr Kubernetes-Cluster diesen Zustand jederzeit widerspiegelt — automatisch, nachvollziehbar und prüfbar.

Das Problem: Kubernetes-Deployments ohne Kontrolle

Um Argo CD zu verstehen, müssen wir zuerst das Problem verstehen, das es löst. Wie sahen Kubernetes-Deployments vor GitOps-Tools aus?

Traditioneller Ansatz (problematisch)

  • Entwickler oder Pipelines laufen direkt kubectl apply auf dem Cluster
  • CI-Pipelines haben direkten Schreibzugriff auf Kubernetes — ein massives Sicherheitsrisiko
  • Niemand weiß sicher, welche Manifest-Version wirklich aktiv ist
  • Manuelle Patches (kubectl edit) hinterlassen keine Spur
  • Drift: Was in Git steht, unterscheidet sich von dem, was im Cluster läuft
  • Rollbacks sind fehleranfällig und manuell

GitOps mit Argo CD (löst das Problem)

  • Git ist die einzige Quelle der Wahrheit — kein direkter Cluster-Zugriff mehr nötig
  • CI-Pipelines schreiben nur in Git, nie direkt in Kubernetes
  • Argo CD zieht Änderungen aus Git und wendet sie kontrolliert an (Pull-Modell)
  • Jede Änderung ist als Pull Request sichtbar, prüfbar und genehmigt
  • Drift wird sofort erkannt und angezeigt — automatische Korrektur optional
  • Rollback = ein git revert, das Argo CD sofort ausrollt

Was ist GitOps? Die vier Prinzipien

GitOps ist kein Produkt — es ist ein Betriebsmodell für Cloud-native Systeme, das von Weaveworks 2017 geprägt und später durch die OpenGitOps-Initiative der CNCF formalisiert wurde. Es basiert auf vier Kernprinzipien:

1. Deklarativ

Der gewünschte Systemzustand wird vollständig deklarativ beschrieben — nicht als Abfolge von Befehlen, sondern als Zielzustand. Kubernetes-Manifeste, Helm Charts und Kustomize-Overlays sind Beispiele für deklarative Definitionen.

2. Versioniert und unveränderlich

Git speichert den Zielzustand versioniert und unveränderlich. Jeder Commit ist ein Snapshot. Die vollständige Änderungshistorie ist jederzeit abrufbar — wer hat was wann geändert und warum (PR-Beschreibung).

3. Pull-basiert

Software-Agenten (hier: Argo CD) ziehen Änderungen aus Git, anstatt dass Pipelines sie in den Cluster drücken. Das eliminiert externe Schreibzugriffe auf den Cluster und reduziert die Angriffsoberfläche drastisch.

4. Kontinuierlich reconciliert

Argo CD beobachtet den Cluster-Zustand kontinuierlich. Weicht der Live-Zustand vom gewünschten Zustand in Git ab (Drift), erkennt Argo CD das sofort und kann automatisch oder auf Anforderung korrigieren.

Wie Argo CD architektonisch aufgebaut ist

Argo CD besteht aus mehreren Komponenten, die zusammen den GitOps-Zyklus implementieren. Das Verständnis dieser Architektur ist essenziell für Produktion, Troubleshooting und Skalierung.

Komponente Aufgabe Technisches Detail
API Server Einstiegspunkt für CLI (argocd), Web-UI und CI-Systeme. Verwaltet Authentifizierung, RBAC und App-Status. gRPC / REST; JWT-Auth; TLS-terminiert
Repository Server Klont Git-Repositories, rendert Manifeste aus Helm Charts, Kustomize-Overlays und plain YAML. Liefert das fertige Manifest-Set an den Application Controller. Läuft isoliert; kein direkter Cluster-Zugriff; cacht Repo-Inhalte
Application Controller Herzstück von Argo CD. Vergleicht den Desired State (aus dem Repo Server) mit dem Live State (aus dem Kubernetes API Server) und löst Syncs aus. Kubernetes-Controller (Reconciliation Loop); schreibt Application-Status-CRDs
Redis Cache für Cluster-Zustände, Repo-Inhalte und berechnete Diffs. Reduziert Last auf API Server und Repo Server. In-Memory; kein persistenter Speicher nötig
Dex (optional) OIDC-Proxy für Single Sign-On (SSO). Verbindet Argo CD mit externen Identity Providern (GitHub, GitLab, LDAP, SAML). Separat deploybar; für Produktion empfohlen
Hinweis zur Skalierung: In großen Umgebungen (50+ Cluster) lässt sich der Application Controller horizontal mit ARGOCD_CONTROLLER_REPLICAS skalieren. Jede Instanz übernimmt dabei eine Teilmenge der zu verwaltenden Application-Objekte.

Argo CD ist nicht das einzige GitOps-Tool. Es ist wichtig, die Unterschiede zu kennen, um für Ihr Team die richtige Wahl zu treffen — oder um in Interviews und Projekten fundiert argumentieren zu können.

1. GitOps sauber verstehen

GitOps bedeutet: Der gewünschte Zustand Ihrer Plattform liegt deklarativ in Git. Kubernetes ist nicht mehr der Ort, an dem Sie manuell Änderungen "hineinklicken", sondern der Ort, an dem ein Controller den in Git beschriebenen Zustand kontinuierlich durchsetzt.

Die Arbeitsregel: Jede fachlich relevante Änderung wird als Pull Request sichtbar, geprüft, gemergt und anschließend vom GitOps-Controller ausgerollt.

CI und GitOps-CD sind getrennte Verantwortungen

CI Pipeline

Baut Artefakte, testet Code, scannt Images und publiziert versionierte Container Images. CI darf Kubernetes nicht direkt verändern.

Argo CD

Liest Git, vergleicht Desired State mit Live State und synchronisiert Kubernetes nach definierten Policies.

Damit wird der Deployment-Prozess nachvollziehbar: Wer eine Änderung genehmigt hat, welche Manifestversion aktiv ist und warum ein Rollout ausgelöst wurde, ist in Git und Argo CD sichtbar.

2. Installation und erster Zugriff

Für die Labs benötigen Sie einen Kubernetes-Cluster, eine funktionierende kubectl-Konfiguration und die Argo CD CLI. Lokal reicht ein Kind-, Minikube- oder Docker-Desktop-Cluster. In produktionsnahen Umgebungen pinnen Sie die Argo-CD-Version statt immer stable zu installieren.

Argo CD installieren
kubectl create namespace argocd
kubectl apply -n argocd --server-side --force-conflicts \
  -f https://raw.githubusercontent.com/argoproj/argo-cd/stable/manifests/install.yaml

kubectl -n argocd get pods
kubectl -n argocd get svc

Für eine lokale Lernumgebung reicht Port-Forwarding. In echten Umgebungen setzen Sie Ingress, TLS, SSO und RBAC kontrolliert auf.

UI und CLI Zugriff
kubectl port-forward svc/argocd-server -n argocd 8080:443

argocd admin initial-password -n argocd
argocd login localhost:8080 --insecure
Sicherheitsnotiz: Das initiale Admin-Passwort ist nur für das Bootstrap geeignet. Danach: Admin-Passwort rotieren, SSO aktivieren, lokale Admin-Nutzung einschränken und Rollen sauber definieren.

3. Die erste deklarative Application

Das zentrale Objekt in Argo CD ist die Application. Sie beschreibt, aus welchem Git-Repository Argo CD liest, welcher Pfad relevant ist, in welchen Cluster deployt wird und welche Sync-Regeln gelten.

application-guestbook.yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: guestbook-dev
  namespace: argocd
spec:
  project: default
  source:
    repoURL: https://github.com/argoproj/argocd-example-apps.git
    targetRevision: HEAD
    path: guestbook
  destination:
    server: https://kubernetes.default.svc
    namespace: guestbook-dev
  syncPolicy:
    syncOptions:
      - CreateNamespace=true
Application anwenden und synchronisieren
kubectl apply -f application-guestbook.yaml

argocd app get guestbook-dev
argocd app diff guestbook-dev
argocd app sync guestbook-dev
kubectl -n guestbook-dev get all

Was Sie beobachten sollen

  • OutOfSync: Git enthält Ressourcen, die im Cluster noch nicht existieren oder abweichen.
  • Synced: Live-Zustand und Git-Zustand stimmen nach Argo-CD-Sicht überein.
  • Healthy: Kubernetes meldet die Ressourcen als funktionsfähig, zum Beispiel verfügbare Pods bei einem Deployment.

Eine gute GitOps-Struktur trennt Basisdefinitionen von Umgebungskonfiguration. Das verhindert Kopieren ganzer Deployments und macht Unterschiede zwischen Dev, Staging und Prod sichtbar.

Pragmatische Regel: Für kleine Teams reicht ein GitOps-Repository. Bei großen Plattformen trennen Sie Plattform-Bootstrap, Team-Anwendungen und Shared-Addons, damit Berechtigungen und Reviews nicht vermischt werden.

5. Sync-Strategien bewusst auswählen

Argo CD synchronisiert nicht blind. Sie definieren, ob Sync manuell ausgelöst wird, ob Drift automatisch korrigiert wird und ob Ressourcen, die aus Git entfernt wurden, auch aus Kubernetes gelöscht werden sollen.

StrategieWann sinnvollRisikoKontrolle
Manual SyncNeue Teams, kritische Migrationen, LernumgebungenDrift bleibt bestehen, bis jemand synchronisiertPR Review plus manuelle Freigabe in Argo CD
Auto-SyncStabile Services mit guter TestpipelineFehlerhafte Git-Änderungen rollen schneller ausBranch Protection, Reviews, CI Gates
PruneGit soll wirklich Quelle der Wahrheit seinGelöschte Manifeste entfernen Live-RessourcenNur mit Review und klarer Ownership aktivieren
Self-HealManuelle Cluster-Änderungen sollen korrigiert werdenNotfall-Änderungen werden zurückgedrehtBreak-glass-Prozess dokumentieren
Produktionsnahe Sync Policy
syncPolicy:
  automated:
    enabled: true
    prune: true
    selfHeal: true
    allowEmpty: false
  syncOptions:
    - CreateNamespace=true

Interpretation der Felder

  • enabled: Aktiviert automatisches Synchronisieren explizit.
  • prune: Entfernt Live-Ressourcen, die nicht mehr in Git definiert sind.
  • selfHeal: Korrigiert manuelle Drift im Cluster zurück auf Git.
  • allowEmpty: Sollte nur bewusst auf true gesetzt werden, wenn eine leere Zielmenge gewollt ist.
Rollback-Falle: Wenn Auto-Sync aktiv ist, müssen Sie vor einem manuellen Rollback die automatische Synchronisierung kontrolliert deaktivieren oder die gewünschte Version in Git wiederherstellen. Sonst setzt Argo CD den neuesten Git-Zustand erneut durch.

6. Sync Waves und Hooks

Nicht jede Ressource darf gleichzeitig angewendet werden. CRDs müssen vor Custom Resources existieren, Namespaces vor Workloads, Migrationen vor neuen Pods. Argo CD löst das mit Hooks und Sync Waves.

Namespace zuerst, Workload danach
apiVersion: v1
kind: Namespace
metadata:
  name: payments-prod
  annotations:
    argocd.argoproj.io/sync-wave: "-2"
---
apiVersion: apps/v1
kind: Deployment
metadata:
  name: payments-api
  namespace: payments-prod
  annotations:
    argocd.argoproj.io/sync-wave: "0"
spec:
  replicas: 3
PreSync Hook für Datenbankmigration
apiVersion: batch/v1
kind: Job
metadata:
  name: payments-db-migrate
  namespace: payments-prod
  annotations:
    argocd.argoproj.io/hook: PreSync
    argocd.argoproj.io/hook-delete-policy: BeforeHookCreation
spec:
  template:
    spec:
      restartPolicy: Never
      containers:
        - name: migrate
          image: ghcr.io/koffi-academy/payments-migrator:1.4.2
          command: ["./migrate", "up"]
Reihenfolge: Argo CD sortiert Synchronisationen unter anderem nach Phase, Wave, Kubernetes-Kind und Namen. Niedrigere Waves laufen vor höheren Waves; negative Werte sind für frühe Schritte wie Namespaces oder CRDs nützlich.

7. Governance, Projekte und Secrets

Ein produktives GitOps-System braucht Grenzen. Nicht jedes Team darf in jedes Cluster deployen, nicht jedes Repository darf jede Ressource erzeugen, und Secrets dürfen nicht im Klartext in Git liegen.

AppProject als Sicherheitsgrenze

projects/platform-prod.yaml
apiVersion: argoproj.io/v1alpha1
kind: AppProject
metadata:
  name: platform-prod
  namespace: argocd
spec:
  description: Production applications owned by the platform team
  sourceRepos:
    - https://github.com/koffi-academy/platform-gitops.git
  destinations:
    - server: https://kubernetes.default.svc
      namespace: payments-prod
  clusterResourceWhitelist:
    - group: ""
      kind: Namespace
  namespaceResourceWhitelist:
    - group: apps
      kind: Deployment
    - group: ""
      kind: Service
    - group: networking.k8s.io
      kind: Ingress

Secrets richtig behandeln

  • Keine Klartext-Secrets in Git committen, auch nicht in privaten Repositories.
  • Nutzen Sie External Secrets Operator, Sealed Secrets oder SOPS mit klarer Key-Rotation.
  • Trennen Sie Secret-Definition, Secret-Wert und Zugriffspolitik.
  • Prüfen Sie Pull Requests mit Secret Scanning und Policy-as-Code.
App-of-Apps-Hinweis: Das App-of-Apps-Muster ist mächtig für Bootstrap, gehört aber unter Admin-Kontrolle. Wer den Parent steuert, kann viele Child Applications beeinflussen.

8. Multi-Cluster mit ApplicationSet

Multi-Cluster-GitOps skaliert nicht, wenn jede Application manuell kopiert wird. ApplicationSets erzeugen Applications aus Generatoren, zum Beispiel aus registrierten Clustern mit passenden Labels.

Cluster registrieren
kubectl config get-contexts
argocd cluster add prod-eu-context --name prod-eu
argocd cluster list

kubectl -n argocd get secrets -l argocd.argoproj.io/secret-type=cluster
kubectl -n argocd label secret <cluster-secret-name> environment=prod region=eu-central-1
applicationsets/payments-prod-clusters.yaml
apiVersion: argoproj.io/v1alpha1
kind: ApplicationSet
metadata:
  name: payments-prod-clusters
  namespace: argocd
spec:
  generators:
    - clusters:
        selector:
          matchLabels:
            environment: prod
  template:
    metadata:
      name: "{{name}}-payments"
    spec:
      project: platform-prod
      source:
        repoURL: https://github.com/koffi-academy/platform-gitops.git
        targetRevision: main
        path: apps/payments/overlays/prod
      destination:
        server: "{{server}}"
        namespace: payments-prod
      syncPolicy:
        automated:
          enabled: true
          prune: true
          selfHeal: true
        syncOptions:
          - CreateNamespace=true

Designentscheidung für Multi-Cluster

Ein Control Plane Cluster

Ein Argo CD steuert mehrere Zielcluster. Einfacher Überblick, aber höhere Bedeutung der Argo-CD-Verfügbarkeit.

Argo CD pro Cluster

Starke Isolation und lokale Verantwortung. Mehr Bootstrap-Aufwand und mehr Controller-Instanzen.

Hybrid

Zentrale Plattform-Addons plus lokale Team-Instanzen. Sinnvoll bei großen Organisationen mit klaren Domänen.

10. Praktische Labs

Die Labs sind so aufgebaut, dass Sie Schritt für Schritt von einer lokalen Demo zu einem produktionsnahen Multi-Cluster-Design kommen. Jedes Lab hat ein klares Erfolgskriterium.

45 Minuten

Installieren Sie Argo CD in einem lokalen Kubernetes-Cluster, greifen Sie per Port-Forwarding auf die UI zu und melden Sie sich mit der CLI an.

Lab 2: Erste Application deployen

60 Minuten

Deployen Sie die Guestbook-Anwendung aus dem offiziellen Beispielrepository. Vergleichen Sie vor und nach dem Sync den Argo-CD-Status.

Erfolg: Die Application ist Synced und Healthy.

Lab 3: Drift erzeugen und beheben

45 Minuten

Ändern Sie die Replica-Anzahl direkt im Cluster, beobachten Sie den Drift in Argo CD und aktivieren Sie anschließend Self-Heal.

Drift absichtlich erzeugen
kubectl -n guestbook-dev scale deploy guestbook-ui --replicas=5
argocd app diff guestbook-dev
argocd app get guestbook-dev
Erfolg: Sie können erklären, warum Argo CD OutOfSync meldet und wie Self-Heal den Git-Zustand wiederherstellt.

Lab 4: Kustomize Overlays bauen

90 Minuten

Erstellen Sie eine Base und zwei Overlays für Dev und Prod. Deployen Sie beide Umgebungen als getrennte Applications.

Lab 5: Sync Waves und Migration Hook

75 Minuten

Ordnen Sie Namespace, ConfigMap, Migration Job und Deployment mit Waves. Der Migration Job läuft als PreSync Hook.

Erfolg: Der Hook läuft vor dem Deployment und wird nach definierter Hook-Policy bereinigt.

Lab 6: ApplicationSet für Multi-Cluster

120 Minuten

Registrieren Sie mindestens zwei Cluster oder simulieren Sie Cluster-Secrets in einer Lernumgebung. Erzeugen Sie Applications über einen Cluster Generator.

Erfolg: Neue Cluster mit Label environment=prod erhalten automatisch die passende Application.

11. Abschlussprojekt

Projekt: Produktionsnaher GitOps-Stack

Bauen Sie ein GitOps-Repository, das eine Beispielanwendung in Dev und Prod ausrollt, Governance über AppProjects nutzt und Prod per ApplicationSet auf mehrere Cluster verteilt.

  • Ein Repository mit Base, Overlays, AppProjects, Applications und ApplicationSet.
  • Manual Sync für Dev, Auto-Sync mit Prune und Self-Heal für Prod.
  • Mindestens eine Sync Wave und ein PreSync Hook.
  • Dokumentierter Rollback-Prozess und Troubleshooting-Runbook.
  • Keine Klartext-Secrets in Git.

Bewertungskriterien

Die Repo-Struktur ist für ein Team nachvollziehbar und wiederholbar.
Sync Policies sind begründet und nicht pauschal aktiviert.
AppProjects begrenzen Repositories, Ziel-Namespaces und Ressourcentypen.
ApplicationSet-Templates sind labelbasiert und nicht clusterhart codiert.
