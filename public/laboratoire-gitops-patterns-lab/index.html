Instructions

Progressive delivery avec Argo Rollouts

Vous implémentez des stratégies de déploiement progressives — Canary, Blue-Green et A/B — puis la promotion automatique basée sur des métriques. Ces patterns réduisent le rayon d'impact des pannes et augmentent la confiance dans les déploiements.

Étape 1 · Déploiement Canary

Le canary décale le trafic progressivement vers la nouvelle version. Dans l'onglet Manifests, enregistrez apps/canary/canary-rollout.yaml : 
apiVersion: argoproj.io/v1alpha1
kind: Rollout
metadata:
  name: canary-demo
  namespace: canary
spec:
  replicas: 5
  strategy:
    canary:
      steps:
      - setWeight: 20
      - pause: {duration: 30s}
      - setWeight: 40
      - pause: {duration: 30s}
      - setWeight: 60
      - pause: {duration: 30s}
      - setWeight: 80
      - pause: {duration: 30s}
  selector:
    matchLabels:
      app: canary-demo
  template:
    metadata:
      labels:
        app: canary-demo
    spec:
      containers:
      - name: canary-demo
        image: nginx:1.25
        ports:
        - name: http
          containerPort: 80
---
apiVersion: v1
kind: Service
metadata:
  name: canary-demo
  namespace: canary
spec:
  ports:
  - port: 80
    targetPort: http
  selector:
    app: canary-demo
Appliquez et observez l'état dans le Terminal : 
kubectl apply -f /root/gitops-platform/apps/canary/canary-rollout.yaml
kubectl argo rollouts get rollout canary-demo -n canary

Étape 2 · Déploiement Blue-Green

Le blue-green bascule instantanément du service actif (blue) vers le service preview (green). Enregistrez apps/blue-green/blue-green-rollout.yaml : 
apiVersion: argoproj.io/v1alpha1
kind: Rollout
metadata:
  name: bluegreen-demo
  namespace: production
spec:
  replicas: 3
  selector:
    matchLabels:
      app: bluegreen-demo
  template:
    metadata:
      labels:
        app: bluegreen-demo
    spec:
      containers:
      - name: bluegreen-demo
        image: nginx:1.25
        ports:
        - name: http
          containerPort: 80
  strategy:
    blueGreen:
      activeService: bluegreen-demo-active
      previewService: bluegreen-demo-preview
      autoPromotionEnabled: false
      scaleDownDelaySeconds: 30
---
apiVersion: v1
kind: Service
metadata:
  name: bluegreen-demo-active
  namespace: production
spec:
  ports:
  - port: 80
    targetPort: http
  selector:
    app: bluegreen-demo
---
apiVersion: v1
kind: Service
metadata:
  name: bluegreen-demo-preview
  namespace: production
spec:
  ports:
  - port: 80
    targetPort: http
  selector:
    app: bluegreen-demo
Appliquez et observez : 
kubectl apply -f /root/gitops-platform/apps/blue-green/blue-green-rollout.yaml
kubectl argo rollouts get rollout bluegreen-demo -n production

Étape 3 · A/B testing

L'A/B route des segments d'utilisateurs vers des versions différentes (via en-têtes). Enregistrez apps/ab-testing/ab-rollout.yaml : 
apiVersion: argoproj.io/v1alpha1
kind: Rollout
metadata:
  name: ab-demo
  namespace: staging
spec:
  replicas: 4
  strategy:
    canary:
      canaryService: ab-demo-canary
      stableService: ab-demo-stable
      trafficRouting:
        managedRoutes:
        - name: header-route
      steps:
      - setCanaryScale:
          weight: 50
      - pause: {duration: 2m}
  selector:
    matchLabels:
      app: ab-demo
  template:
    metadata:
      labels:
        app: ab-demo
    spec:
      containers:
      - name: ab-demo
        image: nginx:1.25
        ports:
        - name: http
          containerPort: 80
---
apiVersion: v1
kind: Service
metadata:
  name: ab-demo-stable
  namespace: staging
spec:
  ports:
  - port: 80
    targetPort: http
  selector:
    app: ab-demo
---
apiVersion: v1
kind: Service
metadata:
  name: ab-demo-canary
  namespace: staging
spec:
  ports:
  - port: 80
    targetPort: http
  selector:
    app: ab-demo
Appliquez et vérifiez : 
kubectl apply -f /root/gitops-platform/apps/ab-testing/ab-rollout.yaml
kubectl argo rollouts get rollout ab-demo -n staging

Étape 4 · Déclencher et suivre les rollouts

Déclenchez un nouveau déploiement en changeant l'image, puis observez la montée progressive. Promouvez ensuite le blue-green manuellement : 
kubectl argo rollouts set image canary-demo -n canary canary-demo=nginx:1.26
kubectl argo rollouts get rollout canary-demo -n canary
kubectl argo rollouts set image bluegreen-demo -n production bluegreen-demo=nginx:1.26
kubectl argo rollouts promote bluegreen-demo -n production
kubectl argo rollouts status bluegreen-demo -n production

Étape 5 · Promotion automatique par métriques

Un AnalysisTemplate décide automatiquement de promouvoir ou de rollback selon le taux de succès. Enregistrez rollouts/analysis-templates/success-rate.yaml : 
apiVersion: argoproj.io/v1alpha1
kind: AnalysisTemplate
metadata:
  name: success-rate
  namespace: canary
spec:
  args:
  - name: service-name
  - name: namespace
  metrics:
  - name: success-rate
    interval: 30s
    count: 5
    successCondition: result >= 0.95
    failureLimit: 3
    provider:
      job:
        spec:
          template:
            spec:
              containers:
              - name: curl
                image: curlimages/curl:8.5.0
                command: ["sh","-c","echo 1.0"]
              restartPolicy: Never
          backoffLimit: 0
Appliquez le template, puis listez tous les rollouts : 
kubectl apply -f /root/gitops-platform/rollouts/analysis-templates/success-rate.yaml
kubectl argo rollouts list rollouts -A
