Instructions

Policy-as-Code avec Kyverno

Vous appliquez des politiques Kyverno pour faire respecter la sécurité, la maîtrise des coûts, la conformité et la prévention de la dérive sur toute la plateforme GitOps. Les politiques valident, mutent et génèrent automatiquement les ressources Kubernetes.

Étape 1 · Vérifier et configurer Kyverno

Kyverno est déjà installé. Vérifiez-le et activez le reporting des politiques dans le Terminal : 
kubectl get pods -n kyverno
kubectl create configmap kyverno-metrics -n kyverno --from-literal=namespaces='default,production,staging,canary' || true
kubectl get configmap kyverno-metrics -n kyverno

Étape 2 · Politiques de sécurité

Imposez des limites de ressources et bloquez les conteneurs privilégiés. Enregistrez ces deux ClusterPolicy dans l'onglet Policies : 
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: require-resource-limits
  annotations:
    policies.kyverno.io/title: Require Resource Limits
    policies.kyverno.io/category: Security
    policies.kyverno.io/severity: high
spec:
  validationFailureAction: Enforce
  background: true
  rules:
  - name: check-resource-limits
    match:
      any:
      - resources:
          kinds: [Pod]
    validate:
      message: "All containers must have CPU and memory limits defined"
      pattern:
        spec:
          containers:
          - resources:
              limits:
                memory: "?*"
                cpu: "?*"
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: block-privileged-containers
  annotations:
    policies.kyverno.io/title: Block Privileged Containers
    policies.kyverno.io/category: Security
    policies.kyverno.io/severity: critical
spec:
  validationFailureAction: Enforce
  background: true
  rules:
  - name: check-privileged
    match:
      any:
      - resources:
          kinds: [Pod]
    validate:
      message: "Privileged containers are not allowed"
      pattern:
        spec:
          containers:
          - =(securityContext):
              =(privileged): false
Appliquez les deux, puis testez avec un pod non conforme (doit être bloqué) : 
kubectl apply -f /root/gitops-platform/policies/security/require-resource-limits.yaml
kubectl apply -f /root/gitops-platform/policies/security/block-privileged.yaml
kubectl run test-no-limits --image=nginx:1.25 -n production --dry-run=server

Étape 3 · Politiques d'optimisation des coûts

Plafonnez les ressources et exigez un PodDisruptionBudget pour les déploiements à plusieurs répliques. Enregistrez : 
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: limit-container-resources
  annotations:
    policies.kyverno.io/title: Limit Container Resources
    policies.kyverno.io/category: Cost Optimization
    policies.kyverno.io/severity: medium
spec:
  validationFailureAction: Enforce
  background: true
  rules:
  - name: check-cpu-limit
    match:
      any:
      - resources:
          kinds: [Pod]
    validate:
      message: "Container CPU limit must not exceed 2 cores"
      foreach:
      - list: "request.object.spec.containers"
        deny:
          conditions:
            any:
            - key: "{{ element.resources.limits.cpu }}"
              operator: GreaterThan
              value: "2000m"
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: require-pdb-for-deployments
  annotations:
    policies.kyverno.io/title: Require PodDisruptionBudget
    policies.kyverno.io/category: Reliability
    policies.kyverno.io/severity: medium
spec:
  validationFailureAction: Audit
  background: true
  rules:
  - name: check-pdb-exists
    match:
      any:
      - resources:
          kinds: [Deployment]
    preconditions:
      all:
      - key: "{{ request.object.spec.replicas }}"
        operator: GreaterThanOrEquals
        value: 3
    validate:
      message: "Deployments with 3+ replicas should have a PodDisruptionBudget"
      deny: {}
Appliquez puis listez ces politiques : 
kubectl apply -f /root/gitops-platform/policies/cost/limit-container-resources.yaml
kubectl apply -f /root/gitops-platform/policies/cost/require-pdb.yaml
kubectl get clusterpolicy | grep -E "(limit-container|require-pdb)"

Étape 4 · Politiques de conformité

Imposez des labels standard et une convention de nommage des namespaces. Enregistrez : 
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: require-standard-labels
  annotations:
    policies.kyverno.io/title: Require Standard Labels
    policies.kyverno.io/category: Compliance
    policies.kyverno.io/severity: medium
spec:
  validationFailureAction: Enforce
  background: true
  rules:
  - name: check-required-labels
    match:
      any:
      - resources:
          kinds: [Deployment, Service, ConfigMap]
    validate:
      message: "Resources must have app, owner, and environment labels"
      pattern:
        metadata:
          labels:
            app: "?*"
            owner: "?*"
            environment: "?*"
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: namespace-naming-convention
  annotations:
    policies.kyverno.io/title: Namespace Naming Convention
    policies.kyverno.io/category: Compliance
    policies.kyverno.io/severity: low
spec:
  validationFailureAction: Enforce
  background: false
  rules:
  - name: check-namespace-name
    match:
      any:
      - resources:
          kinds: [Namespace]
    validate:
      message: "Namespace name must start with team-, project-, cluster-, or be a system namespace"
      deny:
        conditions:
          all:
          - key: "{{ request.object.metadata.name }}"
            operator: AllNotIn
            value: ["team-*", "project-*", "cluster-*"]
Appliquez puis vérifiez : 
kubectl apply -f /root/gitops-platform/policies/compliance/require-labels.yaml
kubectl apply -f /root/gitops-platform/policies/compliance/namespace-naming.yaml
kubectl get clusterpolicy | grep -E "(require-standard|namespace-naming)"

Étape 5 · Prévention de la dérive

Exigez une annotation de suivi GitOps en production/staging, et auto-étiquetez les ressources gérées. Enregistrez : 
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: prevent-manual-changes
  annotations:
    policies.kyverno.io/title: Prevent Manual Changes
    policies.kyverno.io/category: GitOps
    policies.kyverno.io/severity: high
spec:
  validationFailureAction: Audit
  background: false
  rules:
  - name: require-gitops-annotation
    match:
      any:
      - resources:
          kinds: [Deployment, StatefulSet, DaemonSet]
          namespaces: [production, staging]
    validate:
      message: "Resources in production/staging must have an argocd tracking-id annotation"
      pattern:
        metadata:
          annotations:
            argocd.argoproj.io/tracking-id: "?*"
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: auto-label-gitops-resources
  annotations:
    policies.kyverno.io/title: Auto-Label GitOps Resources
    policies.kyverno.io/category: GitOps
    policies.kyverno.io/severity: low
spec:
  background: true
  rules:
  - name: add-gitops-labels
    match:
      any:
      - resources:
          kinds: [Deployment, Service, ConfigMap]
    mutate:
      patchStrategicMerge:
        metadata:
          labels:
            +(managed-by): gitops
            +(gitops-platform): koffi
Appliquez, puis affichez toutes les politiques et le rapport de conformité : 
kubectl apply -f /root/gitops-platform/policies/drift-prevention/prevent-manual-changes.yaml
kubectl apply -f /root/gitops-platform/policies/drift-prevention/auto-label-gitops.yaml
kubectl get clusterpolicy
kubectl get policyreport -A
