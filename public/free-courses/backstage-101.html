Was ist Backstage?

Backstage ist eine offene Developer-Portal-Plattform, ursprunglich von Spotify entwickelt und 2020 als Open-Source unter CNCF-Schirmherrschaft veroffentlicht. Im Jahr 2023 erhielt es den CNCF Incubating-Status — ein Zeichen fur Reife, Akzeptanz und eine aktive Gemeinschaft.

Die Kernidee: In wachsenden Engineering-Organisationen verlieren Entwickler den Uberblick uber interne Tools, Services, APIs, Dokumentation und Deployment-Pipelines. Jedes Team nutzt andere Werkzeuge und schreibt Dokumentation an anderen Orten. Backstage lost dieses Problem, indem es alles unter einer einheitlichen Oberflache zusammenfuhrt — einem sogenannten Internal Developer Portal (IDP).

CNCF-Status 2023 (Incubating): Backstage ist das meistgenutzte Open-Source-Developer-Portal. Uber 1.100 Unternehmen — darunter American Airlines, Spotify, IKEA und Zalando — betreiben es in Produktion.

Das Problem: Developer-Portal-Chaos in wachsenden Teams

Ohne Backstage

Services uber Dutzende Wikis, Confluence-Seiten und README-Dateien verteilt. Niemand weiss, wer fur welchen Service verantwortlich ist. Neue Entwickler brauchen Wochen. Dokumentation ist veraltet.

Mit Backstage

Zentrales Portal: alle Services, Ownership, Docs, CI/CD-Status und API-Specs an einem Ort. Onboarding in Stunden statt Wochen. Dokumentation automatisch generiert und immer aktuell.

Die vier Kernkomponenten von Backstage

Software Catalog

Das Herzstuck: ein zentrales Register aller Komponenten, APIs, Ressourcen, Systeme und Gruppen. Jeder Eintrag ist eine catalog-info.yaml-Datei im Repository.

TechDocs

Docs-as-Code: Dokumentation wird aus Markdown im Repository mit MkDocs gerendert und direkt im Portal angezeigt. Immer synchron mit dem Code.

Scaffolder

Software-Templates: Entwickler erstellen neue Services uber ein Formular. Im Hintergrund wird ein Repository angelegt, CI/CD konfiguriert und der Catalog-Eintrag registriert.

Plugin-System

Uber 200 Community-Plugins (GitHub Actions, PagerDuty, Kubernetes, Grafana, SonarQube) machen Backstage zum universellen Hub.

What is Backstage?

Backstage is an open developer portal platform originally created by Spotify and open-sourced in 2020 under CNCF incubation. It received CNCF Incubating status in 2023 — signaling maturity and broad industry adoption with over 1,100 production deployments worldwide.

The core idea: as engineering organizations grow, developers lose track of internal tools, services, APIs, documentation, and pipelines. Each team uses different tools and writes docs in different places. Backstage consolidates everything under one unified interface — an Internal Developer Portal (IDP).

CNCF Incubating (2023): Over 1,100 companies including American Airlines, Spotify, IKEA, and Zalando run Backstage in production.

The Four Core Components

Software Catalog

Central registry of all components, APIs, resources, systems, and groups. Each entry is a catalog-info.yaml file in the repository.

TechDocs

Docs-as-Code: documentation is rendered from Markdown using MkDocs and displayed directly in the portal. Always in sync with the code.

Scaffolder

Software templates: developers create new services via a form. Behind the scenes, a repository is created, CI/CD configured, and the catalog entry registered.

Plugin System

Over 200 community plugins (GitHub Actions, PagerDuty, Kubernetes, Grafana, SonarQube) make Backstage a universal engineering hub.

Modul 1 — Installation & Grundkonfiguration

Backstage wird mit dem offiziellen CLI-Tool @backstage/create-app erstellt. Es generiert eine vollstandige Monorepo-Struktur mit Frontend, Backend und Konfigurationsdateien.

Voraussetzungen

  • Node.js 18 LTS oder 20 LTS
  • Yarn 1.x (Classic) — Backstage nutzt Yarn Workspaces
  • Git, Docker (fur TechDocs-Build)
  • PostgreSQL 13+ (Produktion); SQLite reicht fur lokale Entwicklung
Terminal — Backstage App erstellen
# Backstage App erstellen
npx @backstage/create-app@latest

# Prompted: App name? meine-backstage-app
# Das CLI generiert die Struktur und installiert Abhangigkeiten

cd meine-backstage-app

# Verzeichnisstruktur:
# packages/
#   app/          Frontend (React)
#   backend/      Backend (Node.js)
# app-config.yaml Hauptkonfiguration
# app-config.local.yaml (gitignored) lokale Overrides

app-config.yaml verstehen

app-config.yaml — Grundkonfiguration
app:
  title: Meine Backstage App
  baseUrl: http://localhost:3000

organization:
  name: Meine Firma

backend:
  baseUrl: http://localhost:7007
  listen:
    port: 7007
  database:
    client: better-sqlite3
    connection: ':memory:'
    # Produktion:
    # client: pg
    # connection:
    #   host: ${POSTGRES_HOST}
    #   port: ${POSTGRES_PORT}
    #   user: ${POSTGRES_USER}
    #   password: ${POSTGRES_PASSWORD}

catalog:
  rules:
    - allow: [Component, System, API, Resource, Location, Group, User]
  locations:
    - type: file
      target: ../../catalog-info.yaml
Terminal — Entwicklungsserver starten
yarn dev
# Startet Frontend (Port 3000) und Backend (Port 7007)
# Hot-Reload aktiv

Modul 2 — Software Catalog

Der Software Catalog ist das Kernelement von Backstage. Er ist ein zentrales Verzeichnis aller Software-Entitaten — Services, APIs, Libraries, Websites und mehr. Jede Entitat wird durch eine catalog-info.yaml-Datei im jeweiligen Repository definiert.

Entity-Typen

KindBeschreibungBeispiel
ComponentEine Software-Einheit: Service, Library, Website, Backendpayment-service, auth-lib
APIEine definierte Schnittstelle (OpenAPI, gRPC, GraphQL, Async)payments-v2-api
ResourceInfrastruktur-Ressource (Datenbank, S3, Message Queue)orders-postgres
SystemGruppe zusammengehoriger Komponentenpayment-system
DomainFachlicher Kontext (DDD-Domain)commerce, identity
GroupTeam oder Organisationseinheitteam-payments
UserIndividuelle Personen (oft aus LDAP/GitHub synchronisiert)alice, bob

catalog-info.yaml — Vollstandiges Beispiel

catalog-info.yaml
apiVersion: backstage.io/v1alpha1
kind: Component
metadata:
  name: payment-service
  title: Payment Service
  description: Verarbeitet Zahlungen via Stripe und PayPal
  annotations:
    github.com/project-slug: meine-org/payment-service
    backstage.io/techdocs-ref: dir:.
    pagerduty.com/service-id: P12345
  tags:
    - java
    - payments
    - critical
  links:
    - url: https://grafana.intern/d/payments
      title: Grafana Dashboard
      icon: dashboard
spec:
  type: service
  lifecycle: production
  owner: team-payments
  system: payment-system
  dependsOn:
    - resource:default/orders-postgres
    - component:default/auth-service
  providesApis:
    - payments-v2-api

GitHub Discovery konfigurieren

app-config.yaml — GitHub Discovery
catalog:
  providers:
    github:
      myOrg:
        organization: 'meine-github-org'
        catalogPath: '/catalog-info.yaml'
        filters:
          branch: 'main'

Best Practice: Jedes Team tragt die Verantwortung fur die catalog-info.yaml in seinem Repository. Backstage zieht diese Datei automatisch ein — kein zentrales Maintenance-Team notig.

Modul 3 — TechDocs

TechDocs ist Backstages Docs-as-Code-Losung. Dokumentation wird als Markdown direkt neben dem Code gehalten, mit MkDocs zu statischem HTML gerendert und im Portal angezeigt.

TechDocs-Workflow

1. Dokumentation schreiben

Markdown-Dateien im docs/-Verzeichnis. mkdocs.yml definiert die Struktur. Standard MkDocs-Features: Navigation, Tabs, Admonitions, Codeblocke.

2. Backstage zeigt an

Backstage baut die Docs automatisch und zeigt sie im Portal an — gleiche Navigation, volltext-durchsuchbar. Kein manueller Deploy notig.

mkdocs.yml
site_name: Payment Service Dokumentation
nav:
  - Einleitung: index.md
  - Architektur: architecture.md
  - API-Referenz: api.md
  - Deployment: deployment.md
plugins:
  - techdocs-core
Produktion: In Produktion TechDocs mit externem Speicher (AWS S3, GCS, Azure Blob) konfigurieren. Docs im CI/CD-Build rendern und hochladen — nicht beim Request.

Modul 4 — Scaffolder & Software Templates

Der Backstage Scaffolder automatisiert die Erstellung neuer Services nach Unternehmensstandards. Ein Entwickler fullt ein Formular aus — im Hintergrund wird ein Repository angelegt, CI/CD konfiguriert und der Catalog-Eintrag registriert.

template.yaml — vollstandiges Beispiel
apiVersion: scaffolder.backstage.io/v1beta3
kind: Template
metadata:
  name: springboot-service
  title: Spring Boot Microservice
  description: Erstellt neuen Spring Boot Service mit CI/CD und Catalog-Eintrag
  tags: [java, spring-boot]
spec:
  owner: platform-engineering
  type: service
  parameters:
    - title: Service-Details
      required: [serviceName, teamName]
      properties:
        serviceName:
          title: Service-Name
          type: string
          pattern: '^[a-z][a-z0-9-]{2,30}$'
        teamName:
          title: Team
          type: string
          ui:field: OwnerPicker
  steps:
    - id: fetch
      name: Template herunterladen
      action: fetch:template
      input:
        url: ./skeleton
        values:
          serviceName: ${{ parameters.serviceName }}
          teamName: ${{ parameters.teamName }}
    - id: publish
      name: GitHub Repository erstellen
      action: publish:github
      input:
        allowedHosts: ['github.com']
        repoUrl: github.com?owner=meine-org&repo=${{ parameters.serviceName }}
        defaultBranch: main
    - id: register
      name: Im Catalog registrieren
      action: catalog:register
      input:
        repoContentsUrl: ${{ steps.publish.output.repoContentsUrl }}
        catalogInfoPath: /catalog-info.yaml
  output:
    links:
      - title: Repository
        url: ${{ steps.publish.output.remoteUrl }}
Sicherheitshinweis: Scaffolder-Steps laufen mit konfigurierten GitHub-Tokens. Schranken Sie die Sichtbarkeit von Templates mit RBAC ein.

Modul 5 — Custom Plugins entwickeln

Das Plugin-System macht Backstage zum echten Developer-Hub. Jede Seite in Backstage ist ein Plugin. Die Architektur trennt Frontend- und Backend-Plugins klar voneinander.

Frontend-Plugin

React-Komponenten die eine neue Seite oder Tab-Ansicht hinzufugen. Registrierung in packages/app/src/App.tsx.

Backend-Plugin

Node.js/Express-Module fur REST-APIs. Registrierung in packages/backend/src/index.ts.

Terminal — neues Plugin erstellen
# Frontend-Plugin
yarn new --select plugin
# Prompted: Plugin-ID mein-cost-dashboard

# Backend-Plugin
yarn new --select backend-plugin
# Prompted: Plugin-ID mein-cost-backend

# Erzeugt:
# plugins/mein-cost-dashboard/
# plugins/mein-cost-backend/
plugins/mein-cost-dashboard/src/plugin.ts
import { createPlugin, createRoutableExtension } from '@backstage/core-plugin-api';
import { rootRouteRef } from './routes';

export const meinCostDashboardPlugin = createPlugin({
  id: 'mein-cost-dashboard',
  routes: { root: rootRouteRef },
});

export const MeinCostDashboardPage = meinCostDashboardPlugin.provide(
  createRoutableExtension({
    name: 'MeinCostDashboardPage',
    component: () =>
      import('./components/DashboardPage').then(m => m.DashboardPage),
    mountPoint: rootRouteRef,
  }),
);

Permission Framework (RBAC)

Das Backstage Permission Framework ermoglicht feingranulare Zugriffssteuerung. Permissions werden pro Plugin definiert und zentral ausgewertet.

packages/backend/src/plugins/permission.ts
import { createBackendModule } from '@backstage/backend-plugin-api';
import { policyExtensionPoint } from '@backstage/plugin-permission-node/alpha';
import { isPermission, AuthorizeResult } from '@backstage/plugin-permission-common';
import { catalogEntityDeletePermission } from '@backstage/plugin-catalog-common/alpha';

export default createBackendModule({
  pluginId: 'permission',
  moduleId: 'allow-rules',
  register(reg) {
    reg.registerInit({
      deps: { policy: policyExtensionPoint },
      async init({ policy }) {
        policy.setPolicy({
          async handle({ permission, credentials }) {
            if (isPermission(permission, catalogEntityDeletePermission)) {
              const groups = credentials.principal?.ownershipEntityRefs ?? [];
              if (groups.includes('group:default/platform-engineering')) {
                return { result: AuthorizeResult.ALLOW };
              }
              return { result: AuthorizeResult.DENY };
            }
            return { result: AuthorizeResult.ALLOW };
          },
        });
      },
    });
  },
});

Modul 7 — Produktionsbetrieb & Kubernetes-Deployment

Fur den Produktionsbetrieb empfiehlt sich Containerisierung und Deployment auf Kubernetes. Das offizielle Backstage-Repository enthalt ein optimiertes Multi-Stage Dockerfile.

Terminal — Production Build & Docker
# Production Build
yarn build:backend --config app-config.yaml --config app-config.production.yaml

# Docker-Image bauen
docker build \
  --file packages/backend/Dockerfile \
  --tag backstage:1.0.0 \
  .

# Testen
docker run -it -p 7007:7007 \
  -e POSTGRES_HOST=localhost \
  -e POSTGRES_USER=backstage \
  -e POSTGRES_PASSWORD=secret \
  backstage:1.0.0
k8s/deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: backstage
  namespace: backstage
spec:
  replicas: 2
  selector:
    matchLabels:
      app: backstage
  template:
    metadata:
      labels:
        app: backstage
    spec:
      containers:
        - name: backstage
          image: registry.meine-firma.de/backstage:1.0.0
          ports:
            - containerPort: 7007
          envFrom:
            - secretRef:
                name: backstage-secrets
          readinessProbe:
            httpGet:
              path: /healthcheck
              port: 7007
            initialDelaySeconds: 10
          resources:
            requests:
              cpu: 250m
              memory: 256Mi
            limits:
              cpu: 1000m
              memory: 1Gi
Produktions-Checkliste: PostgreSQL statt SQLite, externe TechDocs-Speicherung (S3/GCS), Auth-Provider konfiguriert, Ingress mit TLS, Secret-Management (Vault), Monitoring (Prometheus unter /metrics), regelmasige Backstage-Updates.
