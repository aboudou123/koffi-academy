Was ist die CNPE-Zertifizierung?

Die Cloud Native Platform Engineer (CNPE)-Zertifizierung ist eine CNCF-Zertifizierung, die Expertise im Bereich Platform Engineering bestatigt. Sie richtet sich an Ingenieure, die interne Entwicklerplattformen (IDPs) entwerfen, bauen und betreiben — die Infrastruktur fur andere Entwickler-Teams.

Die Zertifizierung ist eine performanzbasierte Online-Prufung (90 Minuten, Open Book) mit praktischen Szenarien. Sie deckt den vollstandigen Stack moderner Platform-Arbeit ab: von Kubernetes-Internals uber IaC, Security und Observability bis hin zu Developer Experience.

Zielgruppe: Site Reliability Engineers, Platform Engineers, DevOps Engineers und Cloud Architects mit mindestens 1-2 Jahren Kubernetes-Erfahrung in Produktionsumgebungen.

What is the CNPE Certification?

The Cloud Native Platform Engineer (CNPE) certification is a CNCF certification that validates expertise in Platform Engineering. It targets engineers who design, build, and operate Internal Developer Platforms (IDPs) — the infrastructure for other developer teams.

The certification is a performance-based online exam (90 minutes, open book) with practical scenarios, covering the full stack of modern platform work: from Kubernetes internals through IaC, security and observability to developer experience.

Target audience: Site Reliability Engineers, Platform Engineers, DevOps Engineers and Cloud Architects with at least 1-2 years of Kubernetes experience in production environments.

Domane 1 — Platform Architecture & Design (22%)

Die grosste Domane der Prufung. Sie testet das konzeptionelle Verstandnis von Platform Engineering als Disziplin, IDP-Design-Prinzipien und Kubernetes-Erweiterungsstrategien.

Schlusselkonzepte

Platform as a Product

Eine interne Entwicklerplattform wird wie ein Produkt fur interne Kunden (Entwickler-Teams) behandelt. Roadmap, SLAs, Feedback-Loops, User Research — alles wie im Produktmanagement.

Golden Paths

"Golden Paths" sind empfohlene, vorkonfigurierte Wege fur haufige Aufgaben (neuen Service deployen, Datenbank bereitstellen). Sie reduzieren kognitive Last bei Entwicklern ohne volle Autonomie einzuschranken.

Control Plane vs. Data Plane

Kubernetes selbst ist eine Control Plane. Crossplane, ArgoCD und Backstage erweitern diese Control Plane. Der Data Plane enthalt die tatsachlichen Workloads.

Cognitive Load

Team Topologies: Platform Engineering reduziert die kognitive Last von Stream-aligned Teams. Die Platform abstrahiert Infrastruktur-Komplexitat als einfache Self-Service-APIs.

Kubernetes-Erweiterungsstrategien

ErweiterungBeschreibungTypische Tools
Custom Resource Definitions (CRDs)Eigene Ressourcentypen in Kubernetes definieren. Basis fur alle Kubernetes-Erweiterungen.Crossplane XRDs, Argo CD AppProjects
Admission WebhooksValidieren oder mutieren Ressourcen bei der Erstellung/Anderung. Fur Policy-Enforcement.OPA Gatekeeper, Kyverno, Vault Agent Injector
Custom Controllers / OperatorsErweitern das Kubernetes-Reconcile-Modell um eigene Logik. SDK: controller-runtime.Crossplane, Prometheus Operator, Cert-Manager

Typische Prufungsszenarien: Domane 1

Haufige Fragetypen: Welche Erweiterungsstrategie fur welchen Use Case? (CRD vs. Webhook vs. Operator). Wie unterscheidet sich Platform Engineering von DevOps? Welche Team-Topologie-Muster passen zu welcher Organisationsgrosse?

Domane 2 — Kubernetes Internals (18%)

Diese Domane pruft tiefes Verstandnis der Kubernetes-Architektur: wie Komponenten interagieren, wie der Scheduler Entscheidungen trifft und wie Networking unter der Haube funktioniert.

Control Plane Komponenten

KomponenteFunktionPrufer-Hinweis
kube-apiserverEinziger Einstiegspunkt fur alle Kubernetes-API-Calls. Validierung, Authentifizierung, Autorisierung.Admission Controller-Reihenfolge: Mutating vor Validating
etcdVerteilter Key-Value-Store fur den gesamten Cluster-State. Einzige persistente Komponente.Ohne etcd-Backup kein Disaster Recovery moglich
kube-schedulerWeist Pods Nodes zu basierend auf Ressourcen, Affinitats-Regeln, Taints und Tolerations.Scheduler-Erweiterung: Custom Schedulers oder Scheduling Framework Plugins
kube-controller-managerFuhrt alle eingebauten Controller aus: Deployment, ReplicaSet, Job, Node, Endpoint Controller.Reconcile-Loop: beobachte desired state vs. actual state

Kubernetes Networking

CNI (Container Network Interface)

Standard-Plugin-Interface fur Kubernetes-Networking. Populare CNIs: Calico (NetworkPolicy), Cilium (eBPF, Service Mesh), Flannel (einfach, kein NetworkPolicy).

kube-proxy / iptables

Implementiert Service-Load-Balancing im Kernel via iptables oder IPVS. Cilium kann kube-proxy komplett ersetzen (eBPF-basiert, besser skalierend).

CoreDNS

Kubernetes-interner DNS-Server. Auflosung von Service-Namen (svc.cluster.local). Konfigurierbar via ConfigMap. Erweiterbar mit Plugins.

Network Policies

Firewall-Regeln fur Pods. Standardmasig erlaubt Kubernetes allen Traffic. Network Policies schranken Ingress/Egress ein. Require CNI mit NP-Support (Calico, Cilium).

Domane 4 — Security & Compliance (15%)

Security in Cloud-Native-Umgebungen ist vielschichtig: von Secrets Management uber Admission Control bis hin zu Supply Chain Security.

Schlussel-Sicherheitskonzepte

RBAC

Roles, ClusterRoles, RoleBindings und ClusterRoleBindings. Principle of Least Privilege. ServiceAccount-Token-Projection.

Pod Security

Pod Security Admission (PSA): Privileged, Baseline, Restricted. Ablsung von Pod Security Policies (PSP) seit K8s 1.25.

Secrets Management

Vault, External Secrets Operator (ESO), Sealed Secrets. Kubernetes Secrets sind nur Base64 — keine Verschlusselung!

Network Policies

Zero-Trust Networking: Default Deny All, dann explizit erlaubte Verbindungen. Cilium NetworkPolicy fur L7-Regeln.

Policy as Code

OPA/Gatekeeper oder Kyverno. ConstraintTemplates und Constraints. Mutating Policies fur automatische Defaults.

Supply Chain Security

Sigstore/Cosign fur Image-Signierung. SBOM (Software Bill of Materials). Admission Webhook pruft Signaturen.

External Secrets Operator (ESO)

ESO synchronisiert Secrets aus externen Systemen (Vault, AWS Secrets Manager, Azure Key Vault) als native Kubernetes Secrets. Er ist die beliebteste Alternative zum Vault Agent Injector fur neue Deployments.

Domane 5 — Observability (13%)

Vollstandige Observability besteht aus den drei Saulen: Metrics, Logs und Traces — plus Service Level Objectives (SLOs) als Qualitatsmassstab.

Die drei Saulen der Observability

Metrics

Prometheus: Scraping, PromQL, Alertmanager. Thanos/Cortex: Long-Term-Storage und Multi-Cluster-Aggregation. Grafana: Dashboards, Alerts.

Logs

Loki + Promtail: Gunstig, LogQL. Elasticsearch + Fluentd: Machtiger, aber ressourcenintensiver. OpenTelemetry Collector: Herstellerunabhangig.

Traces

Jaeger: Open Source, CNCF. Tempo: Gunstig, Grafana-integriert. OpenTelemetry: Standard-SDK fur Instrumentierung.

SLOs, SLIs und Error Budgets

SLI (Service Level Indicator)

Eine messbare Metrik: Verfugbarkeit, Latenz (p99), Fehlerrate, Durchsatz. Beispiel: "99,2% aller Requests in unter 200ms beantwortet."

SLO (Service Level Objective)

Zielwert fur einen SLI: "99,9% Verfugbarkeit uber 30 Tage." Das Error Budget ist 100% minus SLO = 0,1% erlaubte Ausfallzeit.

Prufer-Tipp: OpenTelemetry ist der CNCF-Standard fur Instrumentierung und ersetzt herstellerspezifische SDKs. Wissen Sie den Unterschied: Collector, SDK, Exporter und Propagation Context.

Domane 6 — Developer Experience (12%)

Developer Experience (DX) ist die kleinste Domane, aber fur Platform Engineers konzeptuell entscheidend. Sie pruft das Verstandnis von IDPs, Developer-Portalen und wie Plattformen die kognitive Last von Entwicklern reduzieren.

Schlusselkonzepte DX

Internal Developer Portal (IDP)

Backstage als zentraler Einstiegspunkt: Software Catalog, TechDocs, Scaffolder-Templates. Reduziert "cognitive load" durch Standardisierung.

Self-Service Infrastruktur

Entwickler erstellen eigene Datenbanken, Queues, Caches via einfache Formulare oder kubectl-Claims — ohne Ops-Ticket. Crossplane-Claims, Backstage-Templates.

Developer Feedback Loops

Schnelles lokales Testen (Tilt, Skaffold, Dev Containers). Kurze CI-Build-Zeiten. Klare Fehlermeldungen bei Policy-Verletzungen.

Platform Metriken

DORA Metrics: Deployment Frequency, Lead Time for Change, Change Failure Rate, Time to Restore. Gemessen mit Sleuth, LinearB oder manuell.

Interaktive Ubungsfragen

Diese Fragen bilden typische CNPE-Prufungsszenarien ab. Klicken Sie auf eine Antwort — das System zeigt an, ob sie korrekt ist, und erklart die richtige Losung.

1. Ein Entwickler-Team beschwert sich, dass sie zu viele Ops-Tickets aufmachen mussen, um neue Datenbanken zu erhalten. Welche Losung entspricht am besten dem Platform-Engineering-Ansatz?
A Das Ops-Team einzustellen, um schneller auf Tickets zu reagieren
B Ein Self-Service-Interface (z.B. Backstage-Template mit Crossplane-Claim) einrichten, sodass Teams eigenstandig Datenbanken bereitstellen konnen
C Alle Teams auf denselben Datenbankserver zwingen, um die Verwaltung zu vereinfachen
D Terraform-Skripte an Entwickler-Teams ubergeben, damit sie Infrastruktur selbst verwalten
Korrekt: B. Platform Engineering zielt auf Self-Service-Infrastruktur mit definierten APIs. Ein Backstage-Template, das im Hintergrund einen Crossplane-Claim erstellt, gibt Entwicklern Autonomie bei gleichzeitiger Platform-Kontrolle (Standards, Sicherheit, Governance).
2. Sie mochten sicherstellen, dass alle Pods in Ihrem Cluster ein bestimmtes Label haben. Welches Kubernetes-Konzept eignet sich am besten?
A NetworkPolicy
B ResourceQuota
C Mutating Admission Webhook (z.B. Kyverno Generate/Mutate)
D RBAC ClusterRole
Korrekt: C. Ein Mutating Admission Webhook (z.B. eine Kyverno-Mutate-Policy) kann Labels automatisch zu Pods hinzufugen, die beim API-Server erstellt werden. NetworkPolicy steuert Netzwerk-Traffic, ResourceQuota begrenzt Ressourcen, RBAC steuert Zugriffsrechte.
3. Was ist der Hauptunterschied zwischen einem Crossplane Managed Resource (MR) und einem Composite Resource Claim (XRC)?
A MRs werden in Git gespeichert, XRCs nicht
B XRCs erfordern eine Providerconfig, MRs nicht
C MRs sind Cluster-scoped, XRCs sind Namespace-scoped und eine Abstraktion fur End-User
D XRCs sind Namespace-scoped Abstraktion fur Entwickler; MRs sind direkte 1:1-Abbildungen realer Cloud-Ressourcen und Cluster-scoped
Korrekt: D. Managed Resources (MRs) sind direkte 1:1-Abbildungen einer Cloud-Ressource (z.B. aws RDSInstance) und immer Cluster-scoped. Composite Resource Claims (XRCs) sind Namespace-scoped Abstraktionen fur Entwickler-Teams — sie deklarieren "ich mochte eine PostgreSQL-Datenbank" ohne die zugrundeliegende Implementierung zu kennen.
4. Welche Aussage uber Kubernetes Secrets ist korrekt?
A Kubernetes Secrets sind standardmasig AES-256 verschlusselt
B Kubernetes Secrets sind nur Base64-kodiert, nicht verschlusselt — ohne zusatzliche Konfiguration (Encryption at Rest, Vault)
C Kubernetes Secrets sind sicherer als ConfigMaps, weil sie in einem separaten etcd-Namespace gespeichert werden
D Nur Cluster-Admins konnen Secrets lesen — andere Rollen haben keinen Zugriff
Korrekt: B. Kubernetes Secrets sind standardmasig nur Base64-kodiert — das ist keine Verschlusselung. Wer etcd-Zugriff hat, kann alle Secrets lesen. Empfehlung: Encryption at Rest aktivieren UND externe Secrets-Systeme (Vault, AWS Secrets Manager) verwenden.
5. Ein Team mochte ihre Deployment-Frequenz und Change Failure Rate messen. Welchem Framework gehoren diese Metriken an?
A DORA Metrics (DevOps Research and Assessment)
B SPACE Framework
C RED Method (Rate, Errors, Duration)
D USE Method (Utilization, Saturation, Errors)
Korrekt: A. Die 4 DORA-Metriken sind: Deployment Frequency, Lead Time for Change, Change Failure Rate und Time to Restore Service. RED Method (Rate, Errors, Duration) ist fur Service-Metriken. USE Method (Utilization, Saturation, Errors) ist fur Infrastruktur-Metriken. SPACE ist ein neueres Developer-Productivity-Framework.
6. Welcher Kubernetes-Scheduler-Mechanismus verhindert, dass Pods auf einem bestimmten Node platziert werden, es sei denn, sie tolerieren explizit diese Einschrankung?
A Node Affinity
B PodAntiAffinity
C Taints und Tolerations
D ResourceQuota
Korrekt: C. Taints werden auf Nodes gesetzt (z.B. kubectl taint nodes node1 gpu=true:NoSchedule). Pods werden nur dann auf diesem Node platziert, wenn sie die entsprechende Toleration haben. Node Affinity zieht Pods zu bestimmten Nodes an, verhindert aber nicht aktiv die Platzierung anderer Pods.
7. Was ist der Hauptvorteil von Cilium gegenuber dem klassischen kube-proxy mit iptables?
A Cilium unterstutzt NetworkPolicies, iptables nicht
B Cilium verwendet eBPF fur Kernel-Level-Networking: wesentlich bessere Performance, L7-Visibility und Service Mesh ohne Sidecars
C Cilium ist kostenlos, kube-proxy erfordert eine Lizenz
D Cilium funktioniert ohne CNI-Plugin
Korrekt: B. Cilium basiert auf eBPF (extended Berkeley Packet Filter), was Kernel-Level-Networking ohne iptables-Overhead ermoglicht. Vorteile: bessere Skalierung bei Tausenden von Services, L7-Protokoll-Visibility (HTTP, gRPC), Service Mesh ohne Sidecar-Proxy (Cilium Mesh) und vollstandiger kube-proxy-Ersatz.
8. Ein OpenTelemetry Collector ist konfiguriert. Was sind die drei Hauptkomponenten einer Collector-Pipeline?
A Scraper, Transformer, Shipper
B Agent, Gateway, Backend
C Metric, Log, Trace
D Receiver, Processor, Exporter
Korrekt: D. Ein OpenTelemetry Collector besteht aus: Receivers (Daten empfangen: OTLP, Prometheus, Jaeger), Processors (Daten transformieren: Batch, Filter, Attribute) und Exporters (Daten senden: Jaeger, Prometheus, OTLP, Loki). Die drei Signaltypen (Metrics, Logs, Traces) sind separat konfiguriert.
