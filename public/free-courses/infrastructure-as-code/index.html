Infrastructure as Code (IaC)

23.06.2026  |  20 Minuten Lernzeit  |  Einsteiger

Server, Netzwerke und Datenbanken werden heute nicht mehr von Hand zusammengeklickt, sondern als Code beschrieben. Dieser Kurs erklärt Schritt für Schritt, was Infrastructure as Code bedeutet, warum es zum Standard geworden ist und wie der Ablauf mit dem bekanntesten Werkzeug, Terraform, in der Praxis aussieht. Keine Vorkenntnisse nötig – wir definieren jeden Begriff, zeigen Beispiele und veranschaulichen alles mit Diagrammen.

Lernziele dieses Kurses

  • Sie können in einem Satz erklären, was Infrastructure as Code ist und welches Problem es löst.
  • Sie verstehen den Unterschied zwischen deklarativem und imperativem Vorgehen.
  • Sie kennen den Terraform-Workflow: write, init, plan, apply, destroy.
  • Sie verstehen die Kernbegriffe Provider, Resource, State, Variable, Output und Modul.
  • Sie können ein einfaches Beispiel lesen und nachvollziehen.

Was ist Infrastructure as Code?

Infrastructure as Code (IaC) bedeutet, IT-Infrastruktur – also Server, Netzwerke, Datenbanken, Firewalls oder Speicher – nicht mehr manuell über eine Weboberfläche einzurichten, sondern in Textdateien zu beschreiben. Diese Dateien werden wie normaler Programmcode in einem Versionsverwaltungssystem (z. B. Git) gespeichert, überprüft und automatisiert ausgeführt.

Statt also in der AWS-Konsole 30 Buttons zu klicken, um einen Server zu erstellen, schreiben Sie ein paar Zeilen Code. Ein Werkzeug liest diesen Code und erzeugt die Infrastruktur automatisch – jedes Mal exakt gleich.

Definition in einem Satz

Infrastructure as Code ist die Praxis, Infrastruktur durch maschinenlesbare Definitionsdateien zu verwalten und bereitzustellen, anstatt durch manuelle Konfiguration – damit sie wiederholbar, versionierbar und automatisierbar wird.

Warum Infrastructure as Code?

Manuelle Konfiguration funktioniert für einen einzelnen Server. Sobald es jedoch um viele Umgebungen (Entwicklung, Test, Produktion) und mehrere Personen geht, entstehen Probleme: niemand weiß mehr genau, wie ein Server eingerichtet wurde, und „bei mir lief es noch“ wird zum Alltag. IaC löst das mit konkreten Vorteilen:

  • Wiederholbarkeit: Dieselbe Datei erzeugt jedes Mal exakt dieselbe Infrastruktur – kein Drift, keine Überraschungen.
  • Versionierung: Jede Änderung steht in Git. Sie sehen, wer wann was geändert hat, und können jederzeit zurückrollen.
  • Geschwindigkeit: Eine komplette Umgebung entsteht in Minuten statt Tagen.
  • Weniger Fehler: Kein vergessener Haken in einem Menü – die Konfiguration ist vollständig dokumentiert.
  • Zusammenarbeit: Infrastruktur wird wie Code geprüft (Code Review), getestet und freigegeben.

Deklarativ vs. Imperativ

Es gibt zwei grundlegende Arten, einem Computer zu sagen, was er tun soll. Diesen Unterschied zu verstehen ist der Schlüssel zu Terraform.

Imperativ – das „Wie“

Sie beschreiben jeden Schritt einzeln: „Erstelle einen Server, dann installiere X, dann öffne Port 443.“ Wie ein Kochrezept. Klassische Skripte (Bash) arbeiten so.

Deklarativ – das „Was“

Sie beschreiben nur das gewünschte Endergebnis: „Ich will einen Server mit offenem Port 443.“ Das Werkzeug ermittelt selbst die nötigen Schritte. Terraform ist deklarativ.

Der Vorteil des deklarativen Ansatzes: Terraform vergleicht den gewünschten Zustand (Ihr Code) mit dem tatsächlichen Zustand (die reale Cloud) und ändert nur das, was nötig ist. Existiert der Server bereits, wird er nicht erneut erstellt.

Der Terraform-Workflow

Terraform ist heute das bekannteste IaC-Werkzeug. Der Arbeitsablauf besteht aus wenigen, immer gleichen Befehlen:

Die wichtigsten Terraform-Konzepte

Sechs Begriffe reichen, um den meisten Terraform-Code zu verstehen:

Provider

Das Plugin, das mit einer bestimmten Cloud spricht (z. B. aws, azurerm, google). Es übersetzt Ihren Code in API-Aufrufe.

Resource

Ein einzelnes Infrastruktur-Objekt, das Sie erstellen wollen – etwa ein Server, ein Speicher-Bucket oder ein Netzwerk.

State

Eine Datei, in der Terraform sich merkt, was es bereits erstellt hat. So weiß es, was geändert oder gelöscht werden muss.

Variable

Platzhalter für Werte (z. B. Region oder Servergröße), damit derselbe Code für mehrere Umgebungen funktioniert.

Output

Ein Ergebniswert, den Terraform nach dem Erstellen ausgibt – etwa die IP-Adresse des neuen Servers.

Modul

Ein wiederverwendbares Paket aus mehreren Resourcen – wie eine Funktion, die Sie an vielen Stellen einsetzen können.

Praktisches Beispiel: ein S3-Bucket mit Terraform

Sehen wir uns ein minimales, vollständiges Beispiel an. Es legt einen Speicher („S3-Bucket“) in der AWS-Region Frankfurt an. Die Datei ist in HCL geschrieben, der Sprache von Terraform – gut lesbar, auch für Einsteiger.

main.tf
# 1. Welcher Provider (Cloud) wird genutzt?
terraform {
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }
}

# 2. Konfiguration des Providers
provider "aws" {
  region = "eu-central-1"   # Frankfurt
}

# 3. Die gewünschte Resource: ein Speicher-Bucket
resource "aws_s3_bucket" "kurs_demo" {
  bucket = "koffi-academy-iac-demo"

  tags = {
    Projekt   = "IaC-Kurs"
    ManagedBy = "Terraform"
  }
}

Diese Datei lesen Sie so: „Nutze den AWS-Provider in Version 5, verbinde dich mit Frankfurt und stelle sicher, dass ein S3-Bucket mit diesem Namen existiert.“ Um sie auszuführen, genügen vier Befehle im Terminal:

Terminal
terraform init      # Lädt den AWS-Provider herunter
terraform plan      # Zeigt die Vorschau: 1 Bucket wird erstellt
terraform apply     # Erstellt den Bucket wirklich (nach Bestätigung)
terraform destroy   # Entfernt alles wieder sauber

Wichtig zu verstehen: Wenn Sie terraform apply ein zweites Mal ausführen, ohne den Code zu ändern, passiert nichts. Terraform erkennt, dass der gewünschte Zustand bereits erreicht ist. Genau das ist die Stärke des deklarativen Ansatzes.

Best Practices für den Einstieg

Alles in Git

Speichern Sie jede .tf-Datei in Git. So haben Sie Historie, Code Review und Rollback – wie bei echtem Code.

Immer erst plan

Führen Sie nie blind apply aus. Lesen Sie zuerst die Ausgabe von plan und prüfen Sie, was sich ändert.

Keine Geheimnisse im Code

Passwörter und Schlüssel gehören nicht in .tf-Dateien, sondern in Variablen oder einen Secret-Manager.

Fazit

Infrastructure as Code verwandelt Infrastruktur in etwas Lesbares, Prüfbares und Wiederholbares. Sie beschreiben deklarativ das gewünschte Ergebnis, Terraform kümmert sich um den Weg dorthin – über init, plan und apply. Mit den sechs Kernbegriffen (Provider, Resource, State, Variable, Output, Modul) und dem Mut, mit einem kleinen Beispiel anzufangen, haben Sie die Grundlage, um moderne Cloud-Infrastruktur professionell und reproduzierbar zu betreiben.
