Enterprise Secret Management mit HashiCorp Vault Integrationsmuster

Master production-grade secret management durch Integration von HashiCorp Vault in Ihre DevPortal-Plattform. Lernen Sie automatisierte Secret-Injection, dynamische Datenbank-Anmeldeinformationen, PKI-Zertifikatsverwaltung und Kubernetes-Authentifizierungsmuster, die von Netflix, Uber und Airbnb in ihren Produktionsplattformen verwendet werden.

Lektüre

Willkommen zur letzten Woche Ihrer Platform Engineering Bootcamp-Reise! Sie haben 19 Wochen damit verbracht, umfassende Fähigkeiten in den Bereichen Container, Orchestrierung, Infrastructure as Code, Automatisierung, Observability und Service Mesh-Technologien aufzubauen. Letzte Woche haben Sie das Fundament Ihrer DevPortal-Plattform mit drei Kubernetes-Clustern, einem CLI-Tool, Anwendungsvorlagen und GitOps-Workflows erstellt.

Diese Woche - Woche 20 - ist der Punkt, an dem alles zusammenkommt. Sie werden Ihre Plattform von einem funktionierenden Prototypen in eine produktionsreife, Enterprise-grade Internal Developer Platform verwandeln, die Sie stolz in Vorstellungsgesprächen und in Ihrem professionellen Portfolio präsentieren können.

Wir beginnen mit dem vielleicht kritischsten Aspekt von Produktionsplattformen: Enterprise Secret Management.

Warum Vault-Integration für Produktionsplattformen wichtig ist

Denken Sie an die Plattformen, die bei Unternehmen laufen, die Sie bewundern:

  • Netflix: Verwaltet Millionen von Secrets über 100.000+ Microservices
  • Uber: Rotiert Datenbank-Anmeldeinformationen automatisch alle 24 Stunden
  • Airbnb: Verwendet dynamische PKI-Zertifikate für Service-to-Service-Authentifizierung
  • Spotify: Injiziert Secrets zur Laufzeit, ohne sie in Git zu speichern

Diese Unternehmen verwenden Kubernetes Secrets nicht direkt in der Produktion. Sie verwenden HashiCorp Vault, das tief in ihre Plattformen integriert ist. Warum?

Warnung: Kubernetes Secrets sind Base64-kodiert, nicht verschlüsselt. Jeder mit Cluster-Zugriff kann sie dekodieren. Sie werden in etcd gespeichert und in GitOps-Workflows in Git-Repositories committet. Dies ist inakzeptabel für Produktionssysteme, die Kundendaten, Zahlungsinformationen oder regulierte Workloads verarbeiten.

HashiCorp Vault löst diese Probleme mit:

  • Verschlüsselung im Ruhezustand und während der Übertragung
  • Dynamische Secrets, die bei Bedarf generiert werden und automatisch ablaufen
  • Audit-Logging für Compliance (SOC 2, HIPAA, PCI-DSS)
  • Zugriffskontrolle mit feingranularen Richtlinien
  • Secret-Rotation ohne Anwendungsausfall

In Woche 18 haben Sie die Vault-Grundlagen gelernt. Diese Woche werden Sie Vault-Produktionsmuster in Ihre DevPortal-Plattform integrieren.

Diese Architektur bietet:

  • Zentrale Secret-Speicherung über alle Umgebungen hinweg
  • Umgebungsisolierung durch separate Vault-Richtlinien

  • Hohe Verfügbarkeit mit Raft-Konsens
  • Audit-Trail für alle Secret-Zugriffe

Integrationsmuster 1: Kubernetes-Authentifizierung

Die Grundlage der Vault-Integration ist die Möglichkeit für Kubernetes-Cluster, sich sicher bei Vault zu authentifizieren. Dies verwendet Kubernetes Service Account Tokens.

Wie es funktioniert:

  • Jeder Kubernetes-Cluster hat ein eindeutiges Service Account, das in Vault konfiguriert ist
  • Anwendungen verwenden das Service Account Token ihres Pods zur Authentifizierung
  • Vault validiert das Token mit dem Kubernetes-API-Server
  • Vault gewährt Zugriff basierend auf Richtlinien, die an das Service Account gebunden sind
# Kubernetes-Authentifizierungsmethode für Dev-Cluster aktivieren vault auth enable -path=kubernetes-dev kubernetes # Vault für die Kommunikation mit dem Dev-Cluster konfigurieren vault write auth/kubernetes-dev/config \ kubernetes_host="https://dev-cluster-api:6443" \ kubernetes_ca_cert=@/path/to/dev-ca.crt \ token_reviewer_jwt=@/path/to/dev-token.jwt # Rolle für Anwendungen im Dev-Cluster erstellen vault write auth/kubernetes-dev/role/devportal-app \ bound_service_account_names=devportal-app \ bound_service_account_namespaces=default,apps \ policies=devportal-dev-policy \ ttl=1h
Warum dies wichtig ist: Dieses Muster eliminiert die Notwendigkeit, Vault-Token in Kubernetes Secrets zu speichern. Anwendungen authentifizieren sich mit ihrer nativen Kubernetes-Identität.

Beispiel aus der Praxis: Spotify verwendet dieses Muster für 1000+ Microservices in über 50 Kubernetes-Clustern. Jeder Service authentifiziert sich automatisch ohne manuelle Token-Verwaltung.

Integrationsmuster 2: Vault Agent Injector (Sidecar-Muster)

Der Vault Agent Injector ist ein mutierender Kubernetes-Validierungs-Webhook, der automatisch Vault Agent Sidecars in Pods injiziert. Dies ist das beliebteste Muster für die Secret-Injection.

Pod-Erstellungsablauf mit Vault Injector 1. kubectl apply deployment.yaml │ ▼ 2. API-Server empfängt Pod-Erstellung │ ▼ 3. Mutierender Webhook (Vault Injector) fängt ab │ ▼ 4. Injector fügt Init-Container + Sidecar-Container hinzu │ ├─→ Init-Container: │ - Authentifizierung bei Vault │ - Abruf initialer Secrets │ - Schreiben nach /vault/secrets/ │ └─→ Sidecar-Container: - Secrets aktualisiert halten - Verlängerungen handhaben - Aktualisierung bei Ablauf │ ▼ 5. Anwendungscontainer startet │ ▼ 6. Anwendung liest Secrets aus /vault/secrets/ │ ▼ 7. Anwendung läuft mit aktuellen Secrets
apiVersion: apps/v1 kind: Deployment metadata: name: web-app spec: template: metadata: annotations: # Vault Agent Injector aktivieren vault.hashicorp.com/agent-inject: "true" vault.hashicorp.com/role: "devportal-app" # Datenbank-Anmeldeinformationen injizieren vault.hashicorp.com/agent-inject-secret-db: "database/creds/web-app-role" vault.hashicorp.com/agent-inject-template-db: | {{- with secret "database/creds/web-app-role" -}} export DB_USERNAME="{{ .Data.username }}" export DB_PASSWORD="{{ .Data.password }}" {{- end -}} # API-Keys injizieren vault.hashicorp.com/agent-inject-secret-api: "secret/data/web-app/api-keys" vault.hashicorp.com/agent-inject-template-api: | {{- with secret "secret/data/web-app/api-keys" -}} export STRIPE_KEY="{{ .Data.data.stripe_key }}" export SENDGRID_KEY="{{ .Data.data.sendgrid_key }}" {{- end -}} spec: serviceAccountName: devportal-app containers: - name: web-app image: devportal/web-app:v1.0 command: ["/bin/sh", "-c"] args: - source /vault/secrets/db && source /vault/secrets/api && ./start-app.sh

Vorteile:

  • Keine Codeänderungen erforderlich - Anwendungen lesen einfach Dateien
  • Automatische Verlängerung - Sidecar kümmert sich um Token-Aktualisierung
  • Vorlagen-Flexibilität - Secrets können als Umgebungsvariablen, JSON, YAML formatiert werden
  • Secret-Lebenszyklus - Secrets verschwinden, wenn der Pod terminiert

Beispiel aus der Praxis: Netflix verwendet dieses Muster in großem Umfang. Ihre Plattform injiziert täglich automatisch Datenbank-Anmeldeinformationen, API-Keys und TLS-Zertifikate in 100.000+ Container.

Integrationsmuster 3: Dynamische Datenbank-Anmeldeinformationen

Eine der leistungsstärksten Funktionen von Vault ist die Generierung von Datenbank-Anmeldeinformationen bei Bedarf. Anstatt ein Passwort über alle Anwendungsinstanzen hinweg zu teilen, erhält jeder Pod eindeutige Anmeldeinformationen.

AspektTraditionelle statische AnmeldeinformationenVault Dynamische Anmeldeinformationen
AnmeldeinformationsbereichGemeinsames Passwort über alle InstanzenEindeutige Anmeldeinformationen pro Pod
RotationManuell, erfordert AnwendungsneustartsAutomatischer Ablauf und Rotation
PrüfbarkeitSchwierig zu verfolgen, welche Instanz auf was zugegriffen hatVollständiger Audit-Trail des Datenbankzugriffs
AuswirkungsbereichKompromittierte Anmeldeinformationen betreffen alle InstanzenKompromittierung auf einzelne Pods isoliert
# Datenbank-Secrets-Engine aktivieren vault secrets enable database # PostgreSQL-Verbindung konfigurieren vault write database/config/postgres \ plugin_name=postgresql-database-plugin \ allowed_roles="web-app-role" \ connection_url="postgresql://{{username}}:{{password}}@postgres:5432/appdb" \ username="vault-admin" \ password="vault-admin-password" # Rolle für Webanwendung erstellen vault write database/roles/web-app-role \ db_name=postgres \ creation_statements="CREATE ROLE \"{{name}}\" WITH LOGIN PASSWORD '{{password}}' VALID UNTIL '{{expiration}}'; \ GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA public TO \"{{name}}\";" \ default_ttl="1h" \ max_ttl="24h"

Beispiel für generierte Anmeldeinformationen:

{ "username": "v-kubernetes-web-app-role-abc123xyz", "password": "A1B2C3D4E5F6G7H8I9J0", "ttl": 3600 }

Beispiel aus der Praxis: Uber rotiert Produktionsdatenbank-Anmeldeinformationen alle 24 Stunden über 8000+ Microservices hinweg. Dieses Muster reduzierte ihren Auswirkungsbereich für Anmeldeinformationskompromittierungen um 99%.

# PKI-Secrets-Engine für Intermediate CA aktivieren vault secrets enable -path=pki-intermediate pki # Intermediate CA generieren vault write pki-intermediate/intermediate/generate/internal \ common_name="DevPortal Platform CA" \ ttl=43800h # Mit Root CA signieren (offline durchgeführt) # Signiertes Zertifikat importieren vault write pki-intermediate/intermediate/set-signed certificate=@signed-cert.pem # Rolle für Dev-Umgebungszertifikate erstellen vault write pki-intermediate/roles/devportal-dev \ allowed_domains="dev.devportal.internal" \ allow_subdomains=true \ max_ttl=720h \ ttl=720h # Zertifikat ausstellen vault write pki-intermediate/issue/devportal-dev \ common_name="api.dev.devportal.internal" \ ttl=720h

Beispiel aus der Praxis: Airbnb verwendet Vault, um täglich 50.000+ TLS-Zertifikate für ihre Microservices auszustellen. Zertifikate haben 24-Stunden-TTLs und werden automatisch erneuert, was starke Sicherheit mit null operativem Aufwand bietet.

Integrationsmuster 5: Richtlinienbasierte Zugriffskontrolle

Vault-Richtlinien steuern, welche Anwendungen auf welche Secrets zugreifen können. Für Ihre DevPortal-Plattform erstellen Sie Richtlinien pro Umgebung und Anwendung.

Vault-Richtlinienstruktur ├── devportal-admin │ └── Vollständiger Zugriff auf alle Secrets und Vault-Konfiguration │ ├── devportal-dev-policy │ ├── Lesen: secret/data/dev/* │ ├── DB-Anmeldeinfo erstellen: database/creds/*-dev-role │ └── Verweigern: secret/data/stage/*, secret/data/prod/* │ ├── devportal-staging-policy │ ├── Lesen: secret/data/stage/* │ ├── DB-Anmeldeinfo erstellen: database/creds/*-stage-role │ └── Verweigern: secret/data/prod/* │ └── devportal-prod-policy ├── Lesen: secret/data/prod/* ├── DB-Anmeldeinfo erstellen: database/creds/*-prod-role └── Verweigern: secret/data/dev/*, secret/data/stage/*
# Richtlinie für Dev-Umgebung-Web-App # Zweck: Dev-Web-App erlaubt Lesen von Secrets und Generieren von DB-Anmeldeinformationen # Lesen statischer Secrets erlauben path "secret/data/dev/web-app/*" { capabilities = ["read"] } # Generierung von Datenbank-Anmeldeinformationen erlauben path "database/creds/web-app-dev-role" { capabilities = ["read"] } # Ausstellung von PKI-Zertifikaten erlauben path "pki-intermediate/issue/devportal-dev" { capabilities = ["create", "update"] } # Zugriff auf Produktions-Secrets verweigern path "secret/data/prod/*" { capabilities = ["deny"] } # Zugriff auf Admin-Operationen verweigern path "sys/*" { capabilities = ["deny"] }

Dies gewährleistet eine starke Isolation zwischen den Umgebungen. Ein kompromittierter Dev-Cluster kann nicht auf Produktions-Secrets zugreifen.

Integrationsmuster 6: Secret-Versionierung und Rollback

Die KV Version 2 Secrets Engine von Vault bietet Versionierungs- und Rollback-Fähigkeiten. Dies ist kritisch für Produktionsplattformen.

# KV v2 aktivieren vault secrets enable -version=2 -path=secret kv # Secrets mit Versionierung schreiben (Version 1) vault kv put secret/web-app/api-keys \ stripe_key="sk_test_abc123" \ sendgrid_key="SG.abc123" # Version 2 (aktualisiert) vault kv put secret/web-app/api-keys \ stripe_key="sk_live_xyz789" \ sendgrid_key="SG.xyz789" # Version 3 (aktualisiert) vault kv put secret/web-app/api-keys \ stripe_key="sk_live_updated" \ sendgrid_key="SG.updated" # Bestimmte Version lesen vault kv get -version=1 secret/web-app/api-keys # Auf Version 1 zurückrollen vault kv rollback -version=1 secret/web-app/api-keys # Alle Versionen auflisten vault kv metadata get secret/web-app/api-keys # Version dauerhaft löschen vault kv destroy -versions=3 secret/web-app/api-keys

Diese Versionierungsfähigkeit bedeutet, dass Sie schnell auf einen vorherigen Secret-Status zurückrollen können, wenn ein Problem auftritt. In Produktionssystemen ist dies entscheidend für die Incident Response.

Kernpunkte: Vault-Integration für Produktionsplattformen

  • Kubernetes-Authentifizierung: Workload-Identität eliminiert manuelle Token-Verwaltung
  • Vault Agent Injector: Sidecar-Muster injiziert Secrets automatisch ohne Codeänderungen
  • Dynamische Anmeldeinformationen: Pro-Pod-Credentials mit automatischem Ablauf minimieren Sicherheitsrisiken
  • PKI-Automatisierung: Kurzlebige TLS-Zertifikate eliminieren manuelle Zertifikatsverwaltung
  • Richtlinienbasierte Zugriffskontrolle: Feingranulare Richtlinien isolieren Umgebungen
  • Secret-Versionierung: Unveränderliche Versionierung mit Rollback für schnelle Incident Response
