Was ist Terraform?

Terraform ist ein Open-Source-Infrastructure-as-Code-Tool, entwickelt von HashiCorp und 2014 veroffentlicht. Es ermoglicht die Definition, Verwaltung und Versionierung von Cloud-Infrastruktur in einer deklarativen Konfigurationssprache namens HCL (HashiCorp Configuration Language).

Anstatt manuell Server, Netzwerke und Datenbanken in einer Cloud-Konsole anzulegen, beschreiben Sie mit Terraform den gewunschten Endzustand Ihrer Infrastruktur in Textdateien. Terraform berechnet dann die notwendigen Anderungen und setzt diese automatisch um.

Marktfuhrend: Terraform ist das meistverbreitete IaC-Tool weltweit. Es unterstutzt uber 3.000 Provider — von AWS, Azure und GCP bis hin zu Kubernetes, GitHub, Datadog und PagerDuty. 2023 wurde Terraform unter der Business Source License (BSL) relizenziert; OpenTofu ist der offizielle CNCF-Fork unter Apache 2.0.

Das Problem: manuelle Infrastrukturverwaltung

Ohne Terraform

Infrastruktur manuell in Cloud-Konsolen erstellt. Keine Versionierung, kein Audit-Trail. "Works on my account"-Probleme. Unterschiedliche Umgebungen (Dev/Staging/Prod) laufen auseinander. Katastrophale Fehler ohne Rollback-Moglichkeit.

Mit Terraform

Infrastruktur als versionierter Code in Git. Reproducible Builds: gleiche Konfiguration ergibt identische Umgebungen. Plan zeigt Anderungen vor der Ausfuhrung. State verfolgt den aktuellen Zustand. Automatisierbar in CI/CD-Pipelines.

Terraform-Workflow: Die drei Kernbefehle

terraform init

Initialisiert das Arbeitsverzeichnis: ladt Provider-Plugins herunter, konfiguriert das Backend, installiert Module.

terraform plan

Berechnet die Anderungen: was wird erstellt, geanderr oder geloscht. Zeigt den Ausfuhrungsplan — ohne etwas zu verandern.

terraform apply

Setzt den Plan um: erstellt, andert oder loscht Ressourcen gemas dem Plan. Aktualisiert den State.

What is Terraform?

Terraform is an open-source Infrastructure as Code tool developed by HashiCorp and released in 2014. It enables defining, managing, and versioning cloud infrastructure in a declarative configuration language called HCL (HashiCorp Configuration Language).

Instead of manually creating servers, networks, and databases in a cloud console, you describe your desired infrastructure state in text files. Terraform then calculates the necessary changes and applies them automatically.

Market leader: Terraform is the world's most widely used IaC tool, supporting 3,000+ providers — from AWS, Azure and GCP to Kubernetes, GitHub, and Datadog. In 2023 HashiCorp relicensed under BSL; OpenTofu is the official CNCF fork under Apache 2.0.

Modul 1 — Installation & Erster Start

Terraform wird als einzelne Go-Binary ausgeliefert. Keine komplexe Installation, keine Dependencies.

Terminal — Terraform Installation
# Linux / macOS via tfenv (empfohlen)
curl -sSL https://raw.githubusercontent.com/tfutils/tfenv/master/install.sh | bash
tfenv install 1.9.0
tfenv use 1.9.0

# macOS via Homebrew
brew tap hashicorp/tap
brew install hashicorp/tap/terraform

# Windows via Chocolatey
choco install terraform

# Version pruefen
terraform version

Erste Konfiguration: Hello World

main.tf — erstes Terraform-Projekt
terraform {
  required_version = ">= 1.9"
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }
}

provider "aws" {
  region = "eu-central-1"
}

resource "aws_s3_bucket" "mein_bucket" {
  bucket = "mein-erster-terraform-bucket-2025"

  tags = {
    Environment = "dev"
    ManagedBy   = "terraform"
  }
}
Terminal — Workflow
terraform init     # Provider herunterladen
terraform fmt      # Code formatieren
terraform validate # Syntax pruefen
terraform plan     # Aenderungen anzeigen
terraform apply    # Aenderungen umsetzen (bestaetigung erforderlich)
terraform destroy  # Alle Ressourcen loeschen

Modul 2 — HCL-Sprache & Ressourcen

HCL (HashiCorp Configuration Language) ist eine deklarative Sprache, die fur Menschen lesbar und fur Maschinen interpretierbar ist. Sie ist JSON-kompatibel, aber viel ausdrucksstarter.

Grundlegende HCL-Konstrukte

main.tf — HCL-Grundstruktur
# Ressource-Block: typ + name
resource "aws_instance" "web_server" {
  ami           = "ami-0c02fb55956c7d316"  # Amazon Linux 2023
  instance_type = "t3.micro"

  vpc_security_group_ids = [aws_security_group.web_sg.id]

  user_data = <<-EOF
    #!/bin/bash
    yum update -y
    yum install -y httpd
    systemctl start httpd
    echo "Hello from Terraform" > /var/www/html/index.html
  EOF

  tags = {
    Name        = "web-server"
    Environment = var.environment
    ManagedBy   = "terraform"
  }
}

# Data Source: bestehende Ressourcen abfragen
data "aws_ami" "amazon_linux" {
  most_recent = true
  owners      = ["amazon"]

  filter {
    name   = "name"
    values = ["al2023-ami-*-x86_64"]
  }
}

# Locals: berechnete Werte
locals {
  common_tags = {
    Project   = "mein-projekt"
    ManagedBy = "terraform"
    Region    = var.aws_region
  }
  bucket_name = "${var.project_name}-${var.environment}-${random_id.suffix.hex}"
}

Referenzen zwischen Ressourcen

main.tf — Ressourcen-Referenzen
resource "aws_security_group" "web_sg" {
  name_prefix = "web-sg-"
  vpc_id      = aws_vpc.main.id  # Referenz auf andere Ressource

  ingress {
    from_port   = 80
    to_port     = 80
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }

  egress {
    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = ["0.0.0.0/0"]
  }
}

# Output: Werte ausgeben
output "web_server_public_ip" {
  description = "Oeffentliche IP-Adresse des Web-Servers"
  value       = aws_instance.web_server.public_ip
}

output "web_server_url" {
  value = "http://${aws_instance.web_server.public_ip}"
}

Modul 3 — State Management

Der Terraform State (terraform.tfstate) ist die Wahrheitsquelle uber den aktuellen Zustand Ihrer Infrastruktur. Terraform vergleicht den State mit der realen Infrastruktur, um Anderungen zu berechnen.

Wichtig: Der State enthalt sensitive Daten (Passworter, Schlussel). Committen Sie terraform.tfstate NIEMALS in ein Git-Repository. Nutzen Sie immer Remote Backends in Produktionsumgebungen.

State-Befehle

Terminal — State-Verwaltung
# State-Inhalt anzeigen
terraform state list
terraform state show aws_instance.web_server

# Ressource aus State entfernen (ohne zu loeschen)
terraform state rm aws_instance.web_server

# Bestehende Ressource in State importieren
terraform import aws_instance.web_server i-1234567890abcdef0

# State-Datei manuell bearbeiten (gefaehrlich)
terraform state pull > state-backup.json
# ... Datei bearbeiten ...
terraform state push state-backup.json

Remote Backend mit S3 + DynamoDB

backend.tf — S3 Remote Backend
terraform {
  backend "s3" {
    bucket         = "mein-terraform-state-bucket"
    key            = "prod/infrastructure/terraform.tfstate"
    region         = "eu-central-1"
    encrypt        = true

    # State Locking via DynamoDB (verhindert gleichzeitige Aenderungen)
    dynamodb_table = "terraform-state-lock"
  }
}
Produktion: Immer Remote Backend + State Locking. S3 + DynamoDB fur AWS, Azure Blob Storage fur Azure, GCS fur Google Cloud. Terraform Cloud bietet beides out-of-the-box.

Modul 4 — Variablen, Outputs & Expressions

Input-Variablen

variables.tf
variable "aws_region" {
  description = "AWS Region fuer die Deployment"
  type        = string
  default     = "eu-central-1"
}

variable "environment" {
  description = "Deployment-Umgebung: dev, staging, prod"
  type        = string
  validation {
    condition     = contains(["dev", "staging", "prod"], var.environment)
    error_message = "Environment muss dev, staging oder prod sein."

  description = "Anzahl der Web-Server-Instanzen"

Modul 5 — Terraform Module

Module sind wiederverwendbare Infrastruktur-Pakete. Sie ermoglichen die Kapselung von Ressourcen-Gruppen, die standardisiert in verschiedenen Projekten verwendet werden konnen.

Eigenes Modul erstellen

modules/web-server/main.tf
# Modul-Verzeichnisstruktur:
# modules/
#   web-server/
#     main.tf       Ressourcen
#     variables.tf  Input-Variablen
#     outputs.tf    Outputs

resource "aws_launch_template" "web" {
  name_prefix   = "${var.name}-"
  image_id      = var.ami_id
  instance_type = var.instance_type

  network_interfaces {
    associate_public_ip_address = true
    security_groups             = [aws_security_group.web.id]
  }

  user_data = base64encode(var.user_data)
}

resource "aws_autoscaling_group" "web" {
  min_size         = var.min_size
  max_size         = var.max_size
  desired_capacity = var.desired_capacity
  vpc_zone_identifier = var.subnet_ids

  launch_template {
    id      = aws_launch_template.web.id
    version = "$Latest"
  }
}

Modul aus der Terraform Registry nutzen

main.tf — Registry-Modul
# Offizielles VPC-Modul von Terraform Registry
module "vpc" {
  source  = "terraform-aws-modules/vpc/aws"
  version = "5.0.0"

  name = "mein-vpc"
  cidr = "10.0.0.0/16"

  azs             = ["eu-central-1a", "eu-central-1b", "eu-central-1c"]
  private_subnets = ["10.0.1.0/24", "10.0.2.0/24", "10.0.3.0/24"]
  public_subnets  = ["10.0.101.0/24", "10.0.102.0/24", "10.0.103.0/24"]

  enable_nat_gateway = true
  enable_vpn_gateway = true

  tags = local.common_tags
}

# EKS-Cluster mit offiziellem Modul
module "eks" {
  source  = "terraform-aws-modules/eks/aws"
  version = "20.0.0"

  cluster_name    = "mein-cluster"
  cluster_version = "1.30"
  vpc_id          = module.vpc.vpc_id
  subnet_ids      = module.vpc.private_subnets
}

Modul 6 — Remote Backends & Workspaces

Workspaces fur Umgebungstrennung

Terraform Workspaces ermoglichen mehrere State-Dateien in einem einzigen Konfigurationsverzeichnis — ideal fur Dev/Staging/Prod-Trennung.

Terminal — Workspace-Verwaltung
# Workspaces verwalten
terraform workspace list
terraform workspace new staging
terraform workspace select prod
terraform workspace show

# In Konfiguration nutzen
resource "aws_instance" "web" {
  instance_type = terraform.workspace == "prod" ? "t3.large" : "t3.micro"
  tags = {
    Environment = terraform.workspace
  }
}
Best Practice: Fur separate Teams und strikte Zugriffstrennung sind separate Verzeichnisse (und eigene State-Buckets) pro Umgebung oft besser als Workspaces.

Modul 7 — CI/CD Integration

Terraform in CI/CD-Pipelines integrieren bedeutet: automatischer terraform plan bei Pull Requests und terraform apply nach Merge in den Hauptbranch.

.github/workflows/terraform.yml
name: Terraform CI/CD

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

permissions:
  id-token: write
  contents: read
  pull-requests: write

jobs:
  terraform:
    name: Terraform Plan & Apply
    runs-on: ubuntu-latest
    defaults:
      run:
        working-directory: infrastructure/

    steps:
      - uses: actions/checkout@v4

      - name: Configure AWS credentials (OIDC)
        uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: arn:aws:iam::123456789012:role/github-actions-terraform
          aws-region: eu-central-1

      - name: Setup Terraform
        uses: hashicorp/setup-terraform@v3
        with:
          terraform_version: "1.9.0"

      - name: Terraform Init
        run: terraform init

      - name: Terraform Format Check
        run: terraform fmt -check

      - name: Terraform Validate
        run: terraform validate

      - name: Terraform Plan
        id: plan
        run: terraform plan -no-color -out=tfplan
        continue-on-error: true

      - name: Comment Plan on PR
        uses: actions/github-script@v7
        if: github.event_name == 'pull_request'
        with:
          script: |
            github.rest.issues.createComment({
              issue_number: context.issue.number,
              owner: context.repo.owner,
              repo: context.repo.repo,
              body: '## Terraform Plan\n```\n${{ steps.plan.outputs.stdout }}\n```'
            })

      - name: Terraform Apply
        if: github.ref == 'refs/heads/main' && github.event_name == 'push'
        run: terraform apply -auto-approve tfplan
OIDC statt Langzeit-Credentials: Die obige Konfiguration nutzt AWS OIDC-Integration. Der GitHub Actions-Runner erhalt kurzlebige AWS-Credentials ohne dass statische Access Keys in den Secrets gespeichert werden mussen.
