Was ist Crossplane?

Crossplane ist ein Kubernetes-natives Infrastructure-as-Code-Framework, entwickelt von Upbound und 2020 in die CNCF aufgenommen. Im Jahr 2024 erhielt Crossplane den CNCF Graduated-Status — das hochste Reife-Level der CNCF, gleichauf mit Kubernetes, Prometheus und Argo CD.

Die revolutionare Idee von Crossplane: Infrastruktur wird uber die Kubernetes-API verwaltet. Anstatt separate IaC-Tools (Terraform, Pulumi) zu erlernen, deklarieren Sie Cloud-Ressourcen als Kubernetes-Custom-Resources und verwalten sie mit kubectl — genau wie Pods oder Deployments.

CNCF Graduated 2024: Crossplane verwaltet Infrastruktur in uber 200 Unternehmen in Produktion. Mit dem AWS-Provider allein konnen uber 1.000 verschiedene AWS-Ressourcentypen deklarativ verwaltet werden.

Das Kernkonzept: Infrastructure as Kubernetes API

Traditionelles IaC

Separate Tools (Terraform CLI, State-Dateien, HCL-Syntax). Infrastruktur und Applikation in unterschiedlichen Pipelines. Kein nativer Kubernetes-Reconcile-Loop — manueller Apply notig.

Crossplane

Infrastruktur als Kubernetes-Custom-Resources. Ein einziger Reconcile-Loop fur Applikation und Infrastruktur. kubectl apply erstellt Cloud-Ressourcen. GitOps (ArgoCD/Flux) verwaltet Infrastruktur und Applikation gleich.

What is Crossplane?

Crossplane is a Kubernetes-native Infrastructure as Code framework developed by Upbound and accepted into CNCF in 2020. In 2024, Crossplane achieved CNCF Graduated status — the highest maturity level, alongside Kubernetes, Prometheus, and Argo CD.

The revolutionary idea: infrastructure is managed via the Kubernetes API. Instead of learning separate IaC tools (Terraform, Pulumi), you declare cloud resources as Kubernetes Custom Resources and manage them with kubectl — exactly like Pods or Deployments.

CNCF Graduated 2024: Crossplane manages infrastructure in 200+ production deployments. The AWS provider alone can manage 1,000+ different AWS resource types declaratively.

Why You Should Master Crossplane

CNCF Graduated 2024: The only IaC tool with CNCF Graduated status — strong signal for long-term stability.
Platform Engineering: Crossplane is the core tool for Internal Developer Platforms — teams get self-service infrastructure without ops tickets.
GitOps-native: Crossplane and ArgoCD/Flux are natural partners — infrastructure and applications in the same GitOps pipeline.
Continuous reconciliation: Crossplane detects and corrects drift automatically — no manual runs needed.
Course approach: Prerequisites: solid Kubernetes fundamentals. We build incrementally — from first managed resources through compositions to full platform engineering setups with composition functions.

Modul 1 — Installation & Erste Schritte

Terminal — Crossplane via Helm installieren
# Crossplane Helm Repo
helm repo add crossplane-stable https://charts.crossplane.io/stable
helm repo update

# Crossplane in eigenen Namespace installieren
helm install crossplane \
  crossplane-stable/crossplane \
  --namespace crossplane-system \
  --create-namespace \
  --version 1.17.0

# Installation pruefen
kubectl get pods -n crossplane-system
kubectl get crds | grep crossplane

Modul 2 — Provider installieren & konfigurieren

Provider sind Kubernetes-Controller, die spezifische Cloud-Plattformen verwalten. Jeder Provider enthalt eine Sammlung von CRDs (Custom Resource Definitions) fur den jeweiligen Cloud-Provider.

provider-aws.yaml — AWS Provider
apiVersion: pkg.crossplane.io/v1
kind: Provider
metadata:
  name: upbound-provider-aws-s3
spec:
  package: xpkg.upbound.io/upbound/provider-aws-s3:v1.14.0
  packagePullPolicy: IfNotPresent
  revisionActivationPolicy: Automatic
provider-config-aws.yaml — AWS Credentials
# Erst: AWS Credentials als Kubernetes Secret
kubectl create secret generic aws-creds \
  --namespace crossplane-system \
  --from-file=creds=./aws-credentials.txt

# ProviderConfig: Wie authentifiziert sich Crossplane bei AWS?
apiVersion: aws.upbound.io/v1beta1
kind: ProviderConfig
metadata:
  name: default
spec:
  credentials:
    source: Secret
    secretRef:
      namespace: crossplane-system
      name: aws-creds
      key: creds
Tipp: In Produktion IRSA (IAM Roles for Service Accounts) oder Workload Identity statt statischer Credentials verwenden.

Modul 3 — Managed Resources

Managed Resources sind direkte 1:1-Abbildungen von Cloud-Ressourcen als Kubernetes-CRDs. Ein kubectl apply erstellt die reale Cloud-Ressource.

s3-bucket.yaml — S3 Bucket als Managed Resource
apiVersion: s3.aws.upbound.io/v1beta1
kind: Bucket
metadata:
  name: mein-crossplane-bucket
  annotations:
    crossplane.io/external-name: mein-crossplane-bucket-2025
spec:
  forProvider:
    region: eu-central-1
    tags:
      Environment: dev
      ManagedBy: crossplane
  providerConfigRef:
    name: default
Terminal — Status und Events
kubectl apply -f s3-bucket.yaml

# Status verfolgen
kubectl get bucket mein-crossplane-bucket
kubectl describe bucket mein-crossplane-bucket

# Crossplane-Events
kubectl get events --sort-by='.metadata.creationTimestamp'

Modul 4 — Composite Resources & Compositions

Composite Resources (XR) sind Abstraktionen uber mehrere Managed Resources. Sie ermoglichen die Erstellung von wiederverwendbaren Infrastruktur-APIs fur Entwickler-Teams — ohne dass diese die zugrundeliegenden Cloud-Details kennen mussen.

XRD — Composite Resource Definition

xrd-postgresql.yaml
apiVersion: apiextensions.crossplane.io/v1
kind: CompositeResourceDefinition
metadata:
  name: xpostgresqlinstances.platform.example.org
spec:
  group: platform.example.org
  names:
    kind: XPostgreSQLInstance
    plural: xpostgresqlinstances
  claimNames:
    kind: PostgreSQLInstance
    plural: postgresqlinstances
  versions:
    - name: v1alpha1
      served: true
      referenceable: true
      schema:
        openAPIV3Schema:
          type: object
          properties:
            spec:
              type: object
              properties:
                parameters:
                  type: object
                  required: [storageGB, size]
                  properties:
                    storageGB:
                      type: integer
                      minimum: 20
                      maximum: 1000
                    size:
                      type: string
                      enum: [small, medium, large]
                    region:
                      type: string
                      default: eu-central-1

Composition — Implementierung

composition-postgresql-aws.yaml
apiVersion: apiextensions.crossplane.io/v1
kind: Composition
metadata:
  name: xpostgresqlinstances.aws.platform.example.org
  labels:
    provider: aws
spec:
  compositeTypeRef:
    apiVersion: platform.example.org/v1alpha1
    kind: XPostgreSQLInstance
  resources:
    - name: rds-instance
      base:
        apiVersion: rds.aws.upbound.io/v1beta1
        kind: Instance
        spec:
          forProvider:
            region: eu-central-1
            engine: postgres
            engineVersion: "16.1"
            skipFinalSnapshot: true
            publiclyAccessible: false
            dbInstanceClass: db.t3.medium
            allocatedStorage: 20
      patches:
        - type: FromCompositeFieldPath
          fromFieldPath: spec.parameters.storageGB
          toFieldPath: spec.forProvider.allocatedStorage
        - type: FromCompositeFieldPath
          fromFieldPath: spec.parameters.region
          toFieldPath: spec.forProvider.region
        - type: FromCompositeFieldPath
          fromFieldPath: spec.parameters.size
          toFieldPath: spec.forProvider.dbInstanceClass
          transforms:
            - type: map
              map:
                small: db.t3.micro
                medium: db.t3.medium
                large: db.r6g.large

Claim — Entwickler-Sicht

postgresql-claim.yaml — Entwickler erstellt Datenbank
apiVersion: platform.example.org/v1alpha1
kind: PostgreSQLInstance
metadata:
  name: meine-datenbank
  namespace: team-a
spec:
  parameters:
    storageGB: 50
    size: medium
    region: eu-central-1
  compositionSelector:
    matchLabels:
      provider: aws
  writeConnectionSecretToRef:
    name: meine-datenbank-secret
Self-Service: Entwickler kennen nur die einfache PostgreSQLInstance-API. Crossplane und die Composition verwalten den komplexen AWS-Stack darunter. Das Platform-Team kontrolliert die Implementierung zentral.

Modul 5 — Composition Functions

Composition Functions ersetzen oder erganzen die deklarativen Patches in Compositions durch programmatische Logik. Sie sind container-basierte gRPC-Services, die in Go, Python, KCL oder Starlark implementiert werden konnen.

composition-with-functions.yaml
# Funktion installieren
apiVersion: pkg.crossplane.io/v1beta1
kind: Function
metadata:
  name: function-patch-and-transform
spec:
  package: xpkg.upbound.io/crossplane-contrib/function-patch-and-transform:v0.6.0

---
# KCL Function fur programmatische Logik
apiVersion: pkg.crossplane.io/v1beta1
kind: Function
metadata:
  name: function-kcl
spec:
  package: xpkg.upbound.io/crossplane-contrib/function-kcl:v0.10.0

---
# Composition nutzt Functions
apiVersion: apiextensions.crossplane.io/v1
kind: Composition
metadata:
  name: xpostgresqlinstances.functions.aws
spec:
  compositeTypeRef:
    apiVersion: platform.example.org/v1alpha1
    kind: XPostgreSQLInstance
  mode: Pipeline
  pipeline:
    - step: patch-and-transform
      functionRef:
        name: function-patch-and-transform
      input:
        apiVersion: pt.fn.crossplane.io/v1beta1
        kind: Resources
        resources:
          - name: rds-instance
            base:
              apiVersion: rds.aws.upbound.io/v1beta1
              kind: Instance
    - step: auto-ready
      functionRef:
        name: function-auto-ready

Modul 6 — Platform Engineering mit Crossplane

Crossplane als Platform-Control-Plane: Entwickler-Teams erhalten eine einfache Self-Service-API fur Infrastruktur. Das Platform-Team definiert die Abstraktion, Compliance und Defaults.

Crossplane + ArgoCD = vollstandige GitOps-Plattform

argocd-app-infra.yaml — Infrastruktur via ArgoCD
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: team-a-infrastructure
  namespace: argocd
spec:
  project: default
  source:
    repoURL: https://github.com/meine-org/team-a-infra
    targetRevision: main
    path: crossplane/claims/
  destination:
    server: https://kubernetes.default.svc
    namespace: team-a
  syncPolicy:
    automated:
      prune: true
      selfHeal: true
Workflow: Entwickler macht Pull Request mit postgresql-claim.yaml. ArgoCD erkennt die Anderung und wendet den Claim an. Crossplane erstellt die RDS-Instanz in AWS. Das Connection-Secret landet automatisch im Namespace des Teams. Alles auditierbar in Git.

Abschlussprojekt — Internes Infrastructure-API

Bauen Sie eine vollstandige interne Infrastruktur-Plattform mit Crossplane.

  • Crossplane auf Kind/Minikube mit AWS- oder Azure-Provider installieren
  • XRD fur eine PostgreSQL-Datenbank definieren
  • Composition mit mindestens 3 Managed Resources implementieren
  • Einen Claim aus einem Entwickler-Namespace heraus erstellen
  • ArgoCD fur automatisches Sync der Crossplane-Claims konfigurieren
  • Composition Function (function-patch-and-transform) integrieren
