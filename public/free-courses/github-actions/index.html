Einleitung — Was ist CI/CD und warum GitHub Actions?

Stell dir vor, dein Team pushed Code — und 3 Minuten später läuft die neue Version automatisch in Production, alle Tests sind grün, Docker Image ist gebaut und dein Team-Slack-Channel hat eine Erfolgsmeldung erhalten. Kein manuelles Bauen, kein „vergessen die Tests zu laufen“, kein Warten auf das Ops-Team. Das ist CI/CD mit GitHub Actions.

CI — Continuous Integration

Jeder Code-Push triggert automatisch Tests, Linting und Build. Fehler werden sofort sichtbar — bevor sie in den Hauptzweig gelangen.

CD — Continuous Deployment

Nach erfolgreichem CI wird die App automatisch auf die Zielumgebung deployed. Staging und Production werden klar getrennt verwaltet.

GitHub Actions

YAML-basierte Automatisierungsplattform direkt in GitHub. 20.000 vorgefertigte Actions im Marketplace. Kostenlos für public Repos.

Modul 1 — Die Anatomie eines Workflows

Jeder GitHub Actions Workflow ist eine YAML-Datei unter .github/workflows/. Er beschreibt wann (Event), was (Jobs & Steps) und wo (Runner) ausgeführt wird.

yaml.github/workflows/ci.yml — Minimaler Workflow
name: CI Pipeline

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

jobs:
  build-and-test:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Set up JDK 21
        uses: actions/setup-java@v4
        with:
          java-version: '21'
          distribution: 'temurin'

      - name: Run tests
        run: mvn test --no-transfer-progress

Schlüsselkonzepte

on: — Trigger-Events

push, pull_request, schedule, workflow_dispatch (manuell), workflow_call (von anderen Workflows). Mehrere Events kombinierbar.

jobs: — Parallele Einheiten

Jobs laufen standardmäßig parallel. Mit needs: [job-name] wird Reihenfolge erzwungen. Jeder Job startet auf einem frischen Runner.

steps: — Schritte im Job

Laufen sequenziell. Entweder uses: (vorgefertigte Action) oder run: (Shell-Befehl). Fehlgeschlagener Step bricht den Job ab.

runs-on: — Runner

GitHub-hosted: ubuntu-latest, macos-latest, windows-latest. Self-hosted: eigene Maschine mit Runner-Agent, volle Kontrolle über Umgebung.

Modul 2 — Events, Jobs & Runner im Detail

Matrix-Strategie: mehrere Versionen parallel testen

yamlMatrix Build — 3 Java-Versionen gleichzeitig
jobs:
  test:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        java: [17, 21, 22]
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-java@v4
        with:
          java-version: ${{ matrix.java }}
          distribution: temurin
      - run: mvn test

Secrets und Umgebungsvariablen

yamlSecrets und Env-Variablen sicher verwenden
env:
  APP_ENV: production
  DATABASE_URL: ${{ secrets.DATABASE_URL }}

steps:
  - name: Deploy with credentials
    env:
      AWS_ACCESS_KEY_ID:     ${{ secrets.AWS_ACCESS_KEY_ID }}
      AWS_SECRET_ACCESS_KEY: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
    run: ./deploy.sh
Secrets nie im YAML hard-coden! Secrets werden im GitHub-Repository unter Settings → Secrets & Variables → Actions gespeichert. Sie sind in Logs automatisch mit *** maskiert.

Modul 3 — Continuous Integration

Das CI-Ziel: kein fehlerhafter Code landet im Hauptzweig. Die Pipeline ist der „Check vor dem Abflug“ — obligatorisch, nicht optional.

yamlVollständige CI Pipeline — Tests + Coverage + Security
name: CI
on: [push, pull_request]

jobs:
  ci:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Cache Maven packages
        uses: actions/cache@v4
        with:
          path: ~/.m2
          key: maven-${{ hashFiles('**/pom.xml') }}

      - name: Unit & Integration Tests
        run: mvn verify --no-transfer-progress

      - name: Publish Test Report
        uses: dorny/test-reporter@v1
        if: always()
        with:
          name: JUnit Tests
          path: target/surefire-reports/*.xml
          reporter: java-junit

      - name: Vulnerability Scan (Trivy)
        uses: aquasecurity/trivy-action@master
        with:
          scan-type: fs
          severity: CRITICAL,HIGH
          exit-code: 1

Branch Protection Rules

Ohne Branch Protection kann jeder direkt auf main pushen ohne die Pipeline abzuwarten. Aktiviere unter Settings → Branches → Add rule:

Require status checks to pass: Pipeline muss grün sein, bevor ein Merge erlaubt ist.
Require pull request reviews: Mindestens 1 Reviewer muss zustimmen.
Restrict force pushes: Kein git push --force auf main.

Modul 4 — Docker & Containerisierung

Multi-Stage Dockerfile für schlanke Production-Images

dockerfileDockerfile — Multi-Stage Build
# Stage 1: Build
FROM maven:3.9-eclipse-temurin-21 AS builder
WORKDIR /app
COPY pom.xml .
RUN mvn dependency:go-offline -q
COPY src ./src
RUN mvn package -DskipTests -q

# Stage 2: Runtime (schlankes Image ~150MB statt ~600MB)
FROM eclipse-temurin:21-jre-alpine
WORKDIR /app
COPY --from=builder /app/target/*.jar app.jar
EXPOSE 8080
ENTRYPOINT ["java", "-jar", "app.jar"]
yamlDocker Image in der Pipeline bauen und mit Git-SHA taggen
steps:
  - name: Build & tag Docker image
    run: |
      IMAGE_TAG="${{ github.sha }}"
      docker build -t myapp:${IMAGE_TAG} -t myapp:latest .
      echo "IMAGE_TAG=${IMAGE_TAG}" >> $GITHUB_ENV
Best Practice — Git-SHA als Tag: myapp:a3b8c21 macht jedes Deployment vollständig reproduzierbar und auf einen exakten Commit zurückverfolgbar. Vermeide :latest in Production.

Modul 5 — AWS ECR & EC2 Deployment

yamlVollständige CD Pipeline — ECR + EC2 + OIDC
name: Deploy to Production
on:
  push:
    branches: [main]

permissions:
  id-token: write   # OIDC: kein statischer Access Key!
  contents: read

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Configure AWS Credentials (OIDC)
        uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: arn:aws:iam::123456789:role/GitHubActionsRole
          aws-region:     eu-central-1

      - name: Login to Amazon ECR
        uses: aws-actions/amazon-ecr-login@v2

      - name: Build & push image to ECR
        run: |
          ECR="123456789.dkr.ecr.eu-central-1.amazonaws.com"
          TAG="${{ github.sha }}"
          docker build -t $ECR/myapp:$TAG .
          docker push $ECR/myapp:$TAG
          echo "ECR_IMAGE=$ECR/myapp:$TAG" >> $GITHUB_ENV

      - name: Deploy to EC2 via SSH
        run: |
          echo "${{ secrets.EC2_SSH_KEY }}" > key.pem
          chmod 600 key.pem
          ssh -i key.pem -o StrictHostKeyChecking=no ec2-user@${{ secrets.EC2_HOST }} \
            "IMAGE=${{ env.ECR_IMAGE }} docker compose pull app && docker compose up -d app"
OIDC — keine statischen Credentials: Mit OpenID Connect erhält der GitHub-Runner ein temporäres Token und assumt eine IAM Role. Kein Access Key, kein Secret Key, kein Rotation-Problem. Das ist der aktuelle Best-Practice-Standard.

Modul 6 — Feature-Branches & Environments

Mit Environments trennt man Staging von Production klar. Production-Deployments können eine manuelle Freigabe (Approval) erfordern.

yamlStaging + Production mit manueller Freigabe
jobs:
  deploy-staging:
    runs-on: ubuntu-latest
    environment:
      name: staging
      url:  https://staging.meine-app.de
    steps:
      - run: echo "Deploy to staging..."

  deploy-production:
    needs: deploy-staging
    runs-on: ubuntu-latest
    environment:
      name: production    # Reviewers müssen manuell bestätigen
      url:  https://meine-app.de
    steps:
      - run: echo "Deploy to production after approval"

Modul 7 — Rollbacks & Blue-Green Deployment

Rollback-StrategieWieDowntimeKomplexitat
Image-Tag RollbackAltes Docker-Tag neu deployen (myapp:a3b8c21)KurzNiedrig
Git RevertRevert-Commit → neue PipelineMittelNiedrig
Blue-GreenZwei Umgebungen, Trafficswitch per Load Balancer0Hoch
yamlAutomatischer Rollback bei fehlgeschlagenem Health Check
- name: Deploy new version
  id: deploy
  run: |
    ssh $HOST "IMAGE=$NEW_IMAGE docker compose up -d app"

- name: Health Check
  id: health
  run: |
    sleep 15  # Warten bis App gestartet
    curl --fail https://meine-app.de/actuator/health

- name: Rollback if health check fails
  if: failure() && steps.health.outcome == 'failure'
  run: |
    echo "Health check failed! Rolling back to $PREV_IMAGE"
    ssh $HOST "IMAGE=$PREV_IMAGE docker compose up -d app"

Modul 8 — Slack Integration

yamlSlack Benachrichtigung — Erfolg und Fehler
- name: Notify Slack (Success)

  uses: slackapi/slack-github-action@v1.26.0

    channel-id: deploys

    SLACK_BOT_TOKEN: ${{ secrets.SLACK_BOT_TOKEN }}

  if: failure()

    channel-id: alerts

      Pipeline-Link: ${{ github.server_url }}/${{ github.repository }}/actions/runs/${{ github.run_id }}
  env:
    SLACK_BOT_TOKEN: ${{ secrets.SLACK_BOT_TOKEN }}

Modul 9 — Best Practices & Weiterführende Pfade

Best Practice Checkliste

Actions pinnen: actions/checkout@v4 statt @main — verhindert Supply-Chain-Angriffe.
Least-Privilege Permissions: permissions: read-all als Default, nur explizit erhöhen was benötigt wird.
OIDC statt Access Keys: Temporäre Tokens per OpenID Connect für AWS/GCP/Azure.
Cache nutzen: Maven-Dependencies cachen spart 60-80% Build-Zeit.
Jobs parallelisieren: Lint, Test und Security-Scan können als parallele Jobs laufen.
Workflow Reuse: workflow_call zum Erstellen wiederverwendbarer Pipeline-Templates für das ganze Team.
