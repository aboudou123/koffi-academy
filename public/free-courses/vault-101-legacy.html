Was ist HashiCorp Vault?

HashiCorp Vault ist ein Secrets-Management-System zur sicheren Speicherung, dem Zugriff und der Rotation von sensiblen Daten: Passwortern, API-Schlusseln, Datenbank-Credentials, Zertifikaten und Verschlusselungsschlusseln. Vault wurde 2015 von HashiCorp veroffentlicht und ist heute der Industriestandard fur Secrets Management in Cloud-Native-Umgebungen.

Das zentrale Konzept von Vault: Anstatt Secrets fest in Konfigurationsdateien, Umgebungsvariablen oder Code-Repositories zu hardcoden, werden sie zentral in Vault gespeichert. Applikationen authentifizieren sich bei Vault und erhalten kurzlebige, automatisch rotierte Credentials — sogenannte Dynamic Secrets.

Industriestandard: Vault wird von Unternehmen wie GitHub, Cloudflare, Shopify und Zalando in Produktion eingesetzt. Es ist integriert mit Kubernetes, AWS, Azure, GCP und allen grossen CI/CD-Plattformen.

Das Problem: Secrets-Sprawl in modernen Architekturen

Ohne Vault

Datenbankpassworter in Umgebungsvariablen, API-Schlussel in .env-Dateien, Zertifikate manuell verwaltet. Secrets leben ewig ohne Rotation. Ein kompromittierter Server gibt alle Secrets preis. Kein Audit-Trail: wer hat wann auf was zugegriffen?

Mit Vault

Zentrales Secrets-Register mit kurzlebigen Dynamic Credentials. Jede Applikation erhaelt nur Zugriff auf ihre eigenen Secrets. Automatische Rotation: Datenbankpassworter ablaufen nach 1 Stunde. Vollstandiger Audit-Log jedes Zugriffs.

What is HashiCorp Vault?

HashiCorp Vault is a secrets management system for securely storing, accessing, and rotating sensitive data: passwords, API keys, database credentials, certificates, and encryption keys. Released in 2015 by HashiCorp, it is now the industry standard for secrets management in cloud-native environments.

The core concept: instead of hardcoding secrets in config files, environment variables, or repositories, they are stored centrally in Vault. Applications authenticate to Vault and receive short-lived, automatically rotated credentials — called Dynamic Secrets.

Industry standard: Vault is used in production by companies like GitHub, Cloudflare, Shopify, and Zalando. It integrates with Kubernetes, AWS, Azure, GCP, and all major CI/CD platforms.

Why You Should Master Vault

Security compliance: SOC 2, ISO 27001, PCI DSS, and HIPAA require centralized secrets management. Vault is the standard solution.
Zero-Trust security: Vault is a core building block of any Zero Trust architecture — short-lived credentials, explicit authorization, complete audit trail.
Kubernetes essential: In Kubernetes environments, Vault is the standard choice for secret injection via the Vault Agent Injector or Vault Secrets Operator.
Career relevance: Vault skills appear in over 45% of senior DevOps and Platform Engineer job postings.

Modul 1 — Installation & Dev-Modus

Terminal — Vault Installation
# macOS via Homebrew
brew tap hashicorp/tap
brew install hashicorp/tap/vault

# Linux (Ubuntu/Debian)
wget -O- https://apt.releases.hashicorp.com/gpg | gpg --dearmor | sudo tee /usr/share/keyrings/hashicorp-archive-keyring.gpg
echo "deb [signed-by=/usr/share/keyrings/hashicorp-archive-keyring.gpg] https://apt.releases.hashicorp.com $(lsb_release -cs) main" | sudo tee /etc/apt/sources.list.d/hashicorp.list
sudo apt update && sudo apt install vault

# Version pruefen
vault version

Vault im Dev-Modus starten

Terminal — Dev-Server
# Dev-Server starten (nur fuer Entwicklung/Lernen!)
vault server -dev

# In neuem Terminal:
export VAULT_ADDR='http://127.0.0.1:8200'
export VAULT_TOKEN='root'  # Dev-Token

vault status
vault secrets list
vault auth list
Dev-Modus: Der Dev-Modus speichert alle Daten im Arbeitsspeicher — nicht persistent. Beim Neustart sind alle Daten verloren. Nur fur lokales Lernen geeignet, niemals fur Produktion.

Modul 2 — Authentifizierungsmethoden

Auth-Methoden definieren, wie sich Clients bei Vault identifizieren. Jede Auth-Methode gibt nach erfolgreicher Authentifizierung ein Token mit einer oder mehreren Policies zuruck.

AppRole — Service-zu-Service-Authentifizierung

Terminal — AppRole konfigurieren
# AppRole Auth-Methode aktivieren
vault auth enable approle

# AppRole erstellen
vault write auth/approle/role/meine-app \
  token_policies="meine-app-policy" \
  token_ttl=1h \
  token_max_ttl=4h \
  secret_id_ttl=10m \
  secret_id_num_uses=40

# RoleID und SecretID abrufen
vault read auth/approle/role/meine-app/role-id
vault write -f auth/approle/role/meine-app/secret-id

# Mit AppRole einloggen
vault write auth/approle/login \
  role_id="..." \
  secret_id="..."

Kubernetes-Authentifizierung

Terminal — Kubernetes Auth einrichten
# Kubernetes Auth aktivieren
vault auth enable kubernetes

# Vault konfigurieren (aus dem Kubernetes-Cluster heraus)
vault write auth/kubernetes/config \
  kubernetes_host="https://kubernetes.default.svc" \
  kubernetes_ca_cert=@/var/run/secrets/kubernetes.io/serviceaccount/ca.crt

# Rolle erstellen: Kubernetes ServiceAccount auf Vault-Policy mappen
vault write auth/kubernetes/role/meine-app \
  bound_service_account_names=meine-app-sa \
  bound_service_account_namespaces=production \
  policies=meine-app-policy \
  ttl=1h

Modul 3 — Secret Engines

KV v2 — Key-Value Secrets

Terminal — KV v2
# KV v2 aktivieren
vault secrets enable -path=secret kv-v2

# Secret schreiben (mit Versioning)
vault kv put secret/meine-app/database \
  username="admin" \
  password="SuperGeheim123!" \
  host="postgres.intern:5432"

# Secret lesen
vault kv get secret/meine-app/database

# Bestimmte Version lesen
vault kv get -version=2 secret/meine-app/database

# Versionshistorie anzeigen
vault kv metadata get secret/meine-app/database

PKI Secret Engine — Zertifikate

Terminal — PKI Engine
# PKI Engine aktivieren
vault secrets enable pki
vault secrets tune -max-lease-ttl=87600h pki

# Root CA generieren
vault write -field=certificate pki/root/generate/internal \
  common_name="Meine Firma Root CA" \
  ttl=87600h > root_ca.crt

# Intermediate CA
vault secrets enable -path=pki_int pki
vault write pki_int/intermediate/generate/internal \
  common_name="Meine Firma Intermediate CA"

# Zertifikat ausstellen
vault write pki_int/issue/meine-app-role \
  common_name="meine-app.intern" \
  ttl="24h"

Modul 4 — Policies & Zugriffskontrolle

Policies steuern, was ein authentifizierter Client mit seinen Tokens tun darf. Sie sind in HCL geschrieben und folgen dem Prinzip des geringsten Privilegs (Least Privilege).

meine-app-policy.hcl
# Datenbankpassworter lesen (nur lesen, nicht schreiben)
path "secret/data/meine-app/database" {
  capabilities = ["read"]
}

# Alle Secrets unter meine-app/ auflisten
path "secret/metadata/meine-app/*" {
  capabilities = ["list"]
}

# Dynamische DB-Credentials erzeugen
path "database/creds/meine-app-role" {
  capabilities = ["read"]
}

# Eigene Token erneuern
path "auth/token/renew-self" {
  capabilities = ["update"]
}

# Eigene Token-Infos lesen
path "auth/token/lookup-self" {
  capabilities = ["read"]
}
Terminal — Policy deployen
# Policy hochladen
vault policy write meine-app-policy meine-app-policy.hcl

# Policies auflisten
vault policy list

# Policy anzeigen
vault policy read meine-app-policy
Least Privilege: Vault verweigert standardmasig alles — nur explizit erlaubte Pfade und Capabilities sind zuganglich. Starten Sie immer mit minimalen Rechten und erweitem Sie bei Bedarf.

Modul 5 — Dynamic Secrets

Dynamic Secrets sind eines der machtigen Features von Vault. Statt statischer Passworter generiert Vault bei jeder Anfrage kurzlebige, einmalige Credentials — direkt in der Datenbank oder Cloud.

Database Secret Engine

Terminal — PostgreSQL Dynamic Credentials
# Database Engine aktivieren
vault secrets enable database

# PostgreSQL-Verbindung konfigurieren
vault write database/config/meine-postgres \
  plugin_name=postgresql-database-plugin \
  allowed_roles="meine-app-role" \
  connection_url="postgresql://{{username}}:{{password}}@postgres:5432/meindb?sslmode=disable" \
  username="vault-root" \
  password="vault-root-password"

# Dynamische Rolle definieren
vault write database/roles/meine-app-role \
  db_name=meine-postgres \
  creation_statements="CREATE ROLE \"{{name}}\" WITH LOGIN PASSWORD '{{password}}' VALID UNTIL '{{expiration}}'; GRANT SELECT ON ALL TABLES IN SCHEMA public TO \"{{name}}\";" \
  default_ttl="1h" \
  max_ttl="24h"

# Dynamische Credentials abrufen (Vault erstellt PostgreSQL-User)
vault read database/creds/meine-app-role
Ergebnis: Jeder Aufruf von vault read database/creds/meine-app-role erstellt einen einmaligen PostgreSQL-User mit 1 Stunde TTL. Nach Ablauf wird der User automatisch geloscht. Kein statisches Passwort mehr — und keine Moglichkeit, alte Credentials wiederzuverwenden.

Modul 6 — Kubernetes-Integration

Vault Agent Injector

Der Vault Agent Injector ist ein Kubernetes Mutating Admission Webhook. Er injiziert automatisch einen Vault Agent als Sidecar-Container in Pods — ohne dass die Applikation Vault kennen muss.

k8s/deployment.yaml — Vault Agent Injector
apiVersion: apps/v1
kind: Deployment
metadata:
  name: meine-app
spec:
  template:
    metadata:
      annotations:
        vault.hashicorp.com/agent-inject: "true"
        vault.hashicorp.com/role: "meine-app"
        # Secret als Datei injizieren
        vault.hashicorp.com/agent-inject-secret-database.env: "secret/data/meine-app/database"
        # Template fuer das Secret-Format
        vault.hashicorp.com/agent-inject-template-database.env: |
          {{- with secret "secret/data/meine-app/database" -}}
          export DB_USER="{{ .Data.data.username }}"
          export DB_PASS="{{ .Data.data.password }}"
          export DB_HOST="{{ .Data.data.host }}"
          {{- end -}}
    spec:
      serviceAccountName: meine-app-sa
      containers:
        - name: meine-app
          image: meine-app:1.0.0
          command: ["/bin/sh", "-c"]
          args: ["source /vault/secrets/database.env && ./meine-app"]

Vault Secrets Operator (VSO)

Der Vault Secrets Operator ist der modernere Ansatz: Er synchronisiert Vault Secrets als native Kubernetes Secrets und hallt sie automatisch aktuell.

vso-secret.yaml — VaultStaticSecret
apiVersion: secrets.hashicorp.com/v1beta1
kind: VaultStaticSecret
metadata:
  name: meine-app-database
  namespace: production
spec:
  type: kv-v2
  mount: secret
  path: meine-app/database
  destination:
    name: meine-app-database-secret  # K8s Secret-Name
    create: true
  refreshAfter: 30s
  vaultAuthRef: vault-auth

Modul 7 — Produktionsbetrieb

Vault Cluster mit Raft-Backend

vault.hcl — Produktionskonfiguration
storage "raft" {
  path    = "/vault/data"
  node_id = "vault-node-1"

  retry_join {
    leader_api_addr = "https://vault-node-2:8200"
  }
  retry_join {
    leader_api_addr = "https://vault-node-3:8200"
  }
}

listener "tcp" {
  address     = "0.0.0.0:8200"
  tls_cert_file = "/vault/tls/vault.crt"
  tls_key_file  = "/vault/tls/vault.key"
}

# Auto-Unseal mit AWS KMS
seal "awskms" {
  region     = "eu-central-1"
  kms_key_id = "arn:aws:kms:eu-central-1:123456789:key/abc-def"
}

api_addr     = "https://vault-node-1:8200"
cluster_addr = "https://vault-node-1:8201"
ui           = true
Produktions-Checkliste: HA-Cluster mit mindestens 3 Nodes (Raft), Auto-Unseal (AWS KMS, Azure Key Vault oder GCP KMS), TLS fur alle Verbindungen, Audit-Log aktiviert, regelmassige Snapshots des Raft-State, Monitoring (Prometheus-Metriken unter /v1/sys/metrics), Vault-Agents fur Zero-Downtime-Rotation.
